## 1. TS-1: `content` 빈문자열 / `reasoning_content` 빈값

**증상**: `choices[0].message.content`가 `""`. 답이 없는 것처럼 보임.

**원인**: 세 생성모델(qwen3.6-35b-a3b, qwen3.5-9b, gemma-4-e4b) 모두 본문을 `message.content`가 아니라 `message.reasoning_content`로 반환합니다. `/no_think`·`enable_thinking=false`·`reasoning_effort=low`·`max_tokens=4000`은 전부 무효였습니다(실측). 상세 배경은 → [reasoning 폴백](../04_아키텍처_API/02_내부API_인터페이스.md) 참조.

**진단명령**:

```bash
curl -s http://localhost:1234/v1/chat/completions \
  -H "Authorization: Bearer lm-studio" \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen/qwen3.6-35b-a3b","messages":[{"role":"user","content":"연차 휴가 신청 절차를 한 문장으로"}],"max_tokens":256}' \
  | python -c 'import sys,json; m=json.load(sys.stdin)["choices"][0]["message"]; print("content_len=",len(m.get("content") or "")); print("reasoning_len=",len(m.get("reasoning_content") or ""))'
```

`content_len=0`이고 `reasoning_len>0`이면 본 증상이 확정입니다.

**해결**:

1. `content`가 빈값이면 `reasoning_content`로 폴백합니다.
2. 폴백 텍스트에서 `<think>...</think>` 태그를 정제합니다.
3. 동일 로직을 모든 모델 호출에 적용합니다(예외 없음).

```python
import re

def extract_answer(message: dict) -> str:
    text = (message.get("content") or "").strip()
    if not text:
        text = (message.get("reasoning_content") or "").strip()
    # <think>...</think> 정제: 닫힌 블록 제거 후, 남은 여는 태그 기준으로 절단
    text = re.sub(r"<think>.*?</think>", "", text, flags=re.DOTALL)
    if "</think>" in text:
        text = text.split("</think>")[-1]
    text = text.replace("<think>", "").strip()
    return text
```

> 근본 대응은 LMStudio 프리셋의 reasoning 종료 토큰 점검입니다([확인필요] 메뉴 위치). 코드 폴백은 그때까지의 필수 게이트입니다.

## 2. TS-2: `finish_reason=length` 무한사고(12,972자)

**증상**: 한 문장 질문에도 모델이 사고를 멈추지 않고 `finish_reason=length`로 잘립니다. 실측에서 한 문장 질문에 **12,972자**가 생성된 뒤 잘렸습니다.

**원인**: reasoning 모드가 종료 토큰 없이 계속 생성합니다. `max_tokens`를 작게 잡으면 답이 나오기 전에 잘리고, 크게 잡아도 종료 토큰이 없으면 계속 채웁니다.

**진단명령**:

```bash
curl -s http://localhost:1234/v1/chat/completions \
  -H "Authorization: Bearer lm-studio" \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen/qwen3.6-35b-a3b","messages":[{"role":"user","content":"연차 휴가 신청 절차는?"}],"max_tokens":1024}' \
  | python -c 'import sys,json; d=json.load(sys.stdin); print("finish=",d["choices"][0]["finish_reason"]); m=d["choices"][0]["message"]; print("total_len=",len((m.get("content") or "")+(m.get("reasoning_content") or "")))'
```

`finish=length`이면 본 증상입니다.

**해결**:

1. `finish_reason=length`면 `max_tokens`를 **3배**로 늘려 **1회** 재시도합니다.
2. 재시도(2회째)도 `length`면 결과에 경고 라벨을 부착하고 반환합니다(무한 재시도 금지).
3. 근본 대응: LMStudio 프리셋의 reasoning **종료 토큰**을 점검합니다([확인필요] 메뉴 위치).

```python
def generate_with_retry(client, model: str, messages: list, max_tokens: int = 4000) -> dict:
    resp = client.chat.completions.create(model=model, messages=messages, max_tokens=max_tokens)
    if resp.choices[0].finish_reason == "length":
        resp = client.chat.completions.create(model=model, messages=messages, max_tokens=max_tokens * 3)
        if resp.choices[0].finish_reason == "length":
            msg = resp.choices[0].message
            answer = extract_answer(msg.model_dump() if hasattr(msg, "model_dump") else dict(msg))
            return {"text": answer, "warning": "TRUNCATED: finish_reason=length (재시도 2회 모두 잘림)"}
    msg = resp.choices[0].message
    return {"text": extract_answer(msg.model_dump() if hasattr(msg, "model_dump") else dict(msg)), "warning": None}
```

> 주의: ctx 8192 모델(qwen 계열)은 `max_tokens*3`가 ctx를 넘지 않도록 입력 토큰과 합산해 상한을 둡니다. ctx 초과는 [§6](#6-ts-6-컨텍스트-초과-ctx-8192--65536) 참조.

## 3. TS-3: 임베딩 `norm` ≠ 1.0 아님

**증상**: 임베딩 벡터의 L2 노름이 1.0이 아님. 유사도 계산이 어긋나거나 인덱스 품질이 떨어집니다.

**원인**: 기준 모델 `text-embedding-nomic-embed-text-v1.5`는 출력이 **이미 L2 정규화**되어 있어 `norm=1.0`입니다(재정규화 금지). 노름이 1.0에서 벗어났다면 (a) 모델을 교체했는데 새 모델이 정규화하지 않거나, (b) 잘못된 후처리/평균 풀링이 끼어든 경우입니다. BGE-M3로 교체 시 768→1024 차원이 바뀌므로 **노름 재측정이 필수**입니다.

**진단명령**:

```bash
curl -s http://localhost:1234/v1/embeddings \
  -H "Authorization: Bearer lm-studio" \
  -H "Content-Type: application/json" \
  -d '{"model":"text-embedding-nomic-embed-text-v1.5","input":"연차 휴가 신청 절차"}' \
  | python -c 'import sys,json,math; v=json.load(sys.stdin)["data"][0]["embedding"]; n=math.sqrt(sum(x*x for x in v)); print("dim=",len(v)); print("norm=",round(n,6)); print("guard_ok=",abs(n-1.0)<1e-3)'
```

`guard_ok=False`이면 본 증상입니다. 기준 모델은 `dim=768`, `norm≈1.0`이어야 합니다.

**해결**:

1. 노름 가드를 임베딩 경로에 둡니다: `abs(norm-1.0) < 1e-3`이 아니면 명시적으로 처리합니다.
2. **정규화된 모델**(기준 nomic): 재정규화하지 않고 가드만 통과시킵니다. 어긋나면 후처리(풀링/평균)를 의심합니다.
3. **비정규화 모델로 교체한 경우**: 인덱스 적재 전에 1회만 L2 정규화하고, 가드 임계를 그 모델 기준으로 재설정합니다.
4. 차원이 바뀌면(768→1024) 인덱스를 **재생성**합니다. 혼합 차원은 즉시 실패시킵니다.

```python
import numpy as np

def guard_embedding(vec, normalize_if_needed: bool = False):
    v = np.asarray(vec, dtype=np.float32)
    norm = float(np.linalg.norm(v))
    if abs(norm - 1.0) < 1e-3:
        return v  # 이미 정규화됨(nomic 기준). 재정규화 금지.
    if normalize_if_needed and norm > 0:
        return v / norm  # 교체 모델이 비정규화일 때만.
    raise ValueError(f"norm guard 위반: norm={norm:.6f} (기준 모델은 1.0이어야 함)")
```

> 인덱스 차원·재생성 정책은 → [임베딩 모델](../03_DATABASE/02_데이터모델_청크.md), 불변식은 → [시스템 불변식](../99_AI참조/01_실측제약_불변식.md) 참조.

## 4. TS-4: 한국어 검색 실패(정답 3위, 격차 -0.018)

**증상**: '연차 휴가 신청 절차' 질의에 정답 문서(휴가규정)가 **3위**, 오답(출장경비)보다 유사도 격차가 **-0.018**. Recall@1=0입니다.

**원인**: 기준 임베딩(nomic)의 한국어 변별력 약점입니다. `search_document`/`search_query` prefix 효과는 미미했습니다(0.687→0.695). 즉 prefix 튜닝으로는 순위가 뒤집히지 않습니다.

**진단명령**:

```bash
python eval/evaluate.py --queries eval/queries.yaml --metrics recall@1,recall@3,mrr --filter 연차휴가
```

회귀 케이스(연차휴가)에서 `recall@1=0`, 정답 순위가 3위로 나오면 본 증상입니다. 골든셋과 측정 절차는 → [평가 지표](./00_평가하네스_Recall.md) 참조.

**해결**:

1. 1차: 임베딩 모델 교체 후보 **BGE-M3**(1024차원, ctx8192, 한국어 Recall@1 우수-벤더주장)로 교체합니다. 모델 다운로드는 **DECISION**입니다. 교체 시 768→1024 인덱스 재생성 + 노름 재측정([§3](#3-ts-3-임베딩-norm--10-아님))이 필수입니다.
2. 2차: 하이브리드(Dense + BM25, RRF 결합)로 어휘 매칭을 보강합니다.
3. 3차(L3): **BGE-reranker-v2-m3** 리랭커로 상위 후보를 재정렬합니다(로컬 서빙 방식 [확인필요]).
4. 모든 변경은 변수 1개씩 바꿔 정량 지표(Recall@1)로 증명합니다. 목표는 회귀 케이스 Recall@1=1입니다.

> 교체·하이브리드·리랭커 로드맵 순서는 → [../../시작프롬프트.md](../../시작프롬프트.md) 참조.

## 5. TS-5: 무음 잘림(입력 초과)

**증상**: 출력이 소리 없이 짧아지거나 비어 있는데 오류가 안 뜸. 응답 `usage.prompt_tokens=0`.

**원인**: 입력이 모델 ctx를 초과했지만, LMStudio가 `usage.prompt_tokens`를 **0으로 오보고**해서 토큰 기준 가드가 무력화됩니다. 즉 `prompt_tokens` 값만 믿으면 입력 초과를 못 잡습니다.

**진단명령**:

```bash
curl -s http://localhost:1234/v1/chat/completions \
  -H "Authorization: Bearer lm-studio" \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen/qwen3.6-35b-a3b","messages":[{"role":"user","content":"테스트"}],"max_tokens":32}' \
  | python -c 'import sys,json; u=json.load(sys.stdin).get("usage",{}); print("prompt_tokens=",u.get("prompt_tokens"))'
```

`prompt_tokens=0`이 나오면 usage 토큰 보고를 **신뢰하지 마십시오**.

**해결**:

1. `usage.prompt_tokens`에 의존하지 않습니다. 입력 길이를 **문자 수** 또는 **로컬 토크나이저**로 직접 측정합니다.
2. 측정한 입력 길이가 모델 ctx에서 `max_tokens` 여유분을 뺀 상한을 넘으면 **호출 전에** 차단/절단합니다.
3. 문자 기준 보수적 상한은 한국어 1토큰≈1~2자 [추정]를 감안해 모델별로 산정합니다(아래 코드의 `char_budget`).

```python
# 로컬 토크나이저가 있으면 그것을, 없으면 문자 수 기준 보수적 가드
def input_guard(text: str, ctx: int, max_tokens: int, count_tokens=None) -> str:
    reserve = max_tokens + 256  # 출력 + 여유
    if count_tokens is not None:
        n = count_tokens(text)
        if n > ctx - reserve:
            raise ValueError(f"입력 초과: {n} tok > 한도 {ctx - reserve}")
        return text
    # 폴백: 문자 수 기준(한국어 1tok≈1.5자 [추정]로 보수적 환산)
    char_budget = int((ctx - reserve) * 1.5)
    if len(text) > char_budget:
        raise ValueError(f"입력 초과: {len(text)}자 > 한도 {char_budget}자 (prompt_tokens=0 오보고 회피)")
    return text
```

> `usage.prompt_tokens=0` 오보고는 임베딩에서도 동일합니다(불변식). → [시스템 불변식](../99_AI참조/01_실측제약_불변식.md) 참조.

## 6. TS-6: 컨텍스트 초과(ctx 8192 / 65536)

**증상**: 청크 또는 프롬프트가 모델 ctx를 넘어 응답이 잘리거나 오류가 납니다.

**원인**: 모델별 ctx가 다른데 단일 청크 크기로 모든 모델을 호출합니다. 실측 ctx는 아래와 같습니다.

| 모델 | ctx | 비고 |
|------|-----|------|
| qwen/qwen3.6-35b-a3b | 8192 | 35B MoE(실측), parallel=4 |
| qwen/qwen3.5-9b | 8192 | 9B급 [추정] |
| google/gemma-4-e4b | 65536 | 4B급 [추정], 긴 ctx |
| text-embedding-nomic-embed-text-v1.5 | — | 임베딩(768차원) |
| BGE-M3(교체후보) | 8192 | 1024차원 |

**진단명령**:

```bash
# LMStudio가 인지한 모델 목록과 ctx 확인(필드 노출 여부는 빌드마다 다름 [확인필요])
curl -s http://localhost:1234/v1/models \
  -H "Authorization: Bearer lm-studio" \
  | python -c 'import sys,json; [print(m["id"]) for m in json.load(sys.stdin)["data"]]'
```

**해결**:

1. 모델별 ctx를 설정으로 분리합니다(qwen 8192 vs gemma 65536). 가장 짧은 ctx(8192)를 기준으로 청크 크기를 잡으면 모든 모델에 안전합니다.
2. 검색 컨텍스트(top-k 청크) 합계가 `ctx - max_tokens - 여유`를 넘지 않도록 동적으로 top-k를 줄입니다.
3. 입력 측정은 [§5](#5-ts-5-무음-잘림-입력-초과)의 `input_guard`를 재사용합니다(`prompt_tokens` 신뢰 금지).
4. BGE-M3 교체 시 임베딩 ctx도 8192이므로, 긴 청크는 분할합니다.

```python
MODEL_CTX = {
    "qwen/qwen3.6-35b-a3b": 8192,
    "qwen/qwen3.5-9b": 8192,
    "google/gemma-4-e4b": 65536,
}

def fit_context(chunks: list[str], model: str, max_tokens: int, count_tokens=None) -> list[str]:
    ctx = MODEL_CTX[model]
    budget = ctx - max_tokens - 256
    out, used = [], 0
    for c in chunks:
        n = count_tokens(c) if count_tokens else int(len(c) / 1.5)
        if used + n > budget:
            break
        out.append(c); used += n
    return out
```

## 7. TS-7: 외부 API 호출 차단

**증상**: OpenAI/Anthropic/Cohere 등으로 나가는 요청이 실패·타임아웃하거나, 의도치 않게 외부로 데이터가 나갈 위험이 있습니다.

**원인**: SDK `base_url`이 로컬(`http://localhost:1234/v1`)로 고정되지 않았거나, 코드 어딘가에 클라우드 엔드포인트/실키가 남아 있습니다. docs-em은 **외부 클라우드 API 절대 금지**이며 로컬 LMStudio만 사용합니다(불변식).

**진단명령**:

```bash
# 1) base_url이 로컬인지 확인
python -c 'from openai import OpenAI; c=OpenAI(base_url="http://localhost:1234/v1", api_key="lm-studio"); print("base_url=", c.base_url)'

# 2) 코드에 외부 엔드포인트/실키 잔존 여부 스캔
grep -rEn 'api\.openai\.com|api\.anthropic\.com|api\.cohere\.|sk-[A-Za-z0-9]{20,}' src/ scripts/ eval/ || echo "clean: 외부 엔드포인트/실키 없음"
```

`grep`이 무언가를 출력하면 즉시 제거합니다. `base_url`이 `http://localhost:1234/v1`가 아니면 잘못된 구성입니다.

**해결**:

1. 모든 클라이언트를 단일 팩토리로 만들고 `base_url`을 로컬로 고정합니다. `api_key`는 더미 `lm-studio`입니다.
2. CI/사전 커밋에서 위 `grep` 스캔을 강제합니다(외부 엔드포인트/실키 발견 시 실패).
3. 네트워크 격리 환경에서는 로컬 외 모든 아웃바운드가 막혀도 정상 동작해야 합니다.

```python
from openai import OpenAI

LOCAL_BASE_URL = "http://localhost:1234/v1"

def make_client() -> OpenAI:
    return OpenAI(base_url=LOCAL_BASE_URL, api_key="lm-studio")  # 외부 키/엔드포인트 사용 금지
```

> 단일 클라이언트 팩토리와 호출 규약은 → [내부 API](../04_아키텍처_API/02_내부API_인터페이스.md), 금지 사항은 → [시스템 불변식](../99_AI참조/01_실측제약_불변식.md) 참조.
