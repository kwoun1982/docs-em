## 1. 전제와 진입점

모든 운영 명령은 단일 CLI 진입점 `app/cli.py`와 평가 스크립트 `eval/evaluate.py`만 사용합니다. `src.ingest.run` 같은 모듈 직접 호출 표현은 이 문서에서 쓰지 않습니다(현재 코드 없음 단계이므로 인터페이스 계약만 고정).

| 명령 | 용도 |
| --- | --- |
| `python app/cli.py ingest <경로>` | 문서 적재 + 청킹 + 임베딩 + 인덱스 반영 |
| `python app/cli.py search "<질의>"` | 검색만 수행(생성 없음) |
| `python app/cli.py ask "<질의>"` | 검색 + 생성(reasoning 폴백 포함) |
| `python app/cli.py rebuild` | 인덱스 전체 재생성 |
| `python eval/evaluate.py` | 골든셋 평가(Recall@k/MRR/nDCG) |

전제 불변식(상세는 SSOT 참조):

- 임베딩 모델 `text-embedding-nomic-embed-text-v1.5`, 768차원, 출력이 이미 L2 정규화됨(norm=1.0). 재정규화 금지, `abs(norm-1.0)<1e-3` 가드. → [../../시작프롬프트.md](../../시작프롬프트.md) §B 참조.
- 모든 LLM 호출은 로컬 LMStudio `http://localhost:1234/v1`(OpenAI 호환, `api_key='lm-studio'`). 외부 클라우드 API 절대 금지.
- 생성 3종 모두 `message.content` 빈문자열, 답은 `message.reasoning_content`로 옴 → 폴백 필수. 계약 상세는 [../04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md).

> 운영 시작 전 항상 환경 점검을 먼저 수행하십시오. `env_check` 절차는 [../01_시스템구성/02_로컬환경_LMStudio실측.md](../01_시스템구성/02_로컬환경_LMStudio실측.md) 참조.

## 2. 문서 추가 → 재인덱싱

### 2.1 증분 추가(권장)

새 문서만 추가하는 경우 전체 재생성 없이 증분 적재합니다.

```bash
# 0) 사전 점검 — LMStudio 가동/모델 로드/임베딩 차원 확인
python scripts/env_check.py

# 1) 증분 적재 (해당 경로의 신규/변경 파일만)
python app/cli.py ingest data/incoming/

# 2) 검색 스모크 테스트 (생성 없이 검색만)
python app/cli.py search "방금 추가한 문서 핵심 키워드"

# 3) 회귀 평가 — 연차휴가 회귀케이스 포함 골든셋
python eval/evaluate.py
```

증분 적재의 동일성 판정은 인덱스 헤더의 메타(파일 해시·청크 수 등)로 수행합니다. 헤더 스키마는 [../03_DATABASE/01_인덱스_메타데이터_설계.md](../03_DATABASE/01_인덱스_메타데이터_설계.md) 참조.

### 2.2 재인덱싱이 필요한 경우(전체 rebuild)

다음 중 하나라도 해당하면 증분이 아니라 전체 `rebuild`를 수행합니다.

- 청킹 파라미터(문장분리 기준·청크 크기·overlap) 변경
- 임베딩 모델/차원 변경(별도 절차 → §3)
- 벡터스토어 백엔드 교체(LanceDB ↔ ChromaDB, **DECISION**)
- 인덱스 헤더 스키마 버전 불일치(헤더 검증 실패)

```bash
python app/cli.py rebuild
python eval/evaluate.py    # rebuild 후 항상 평가로 동등성/개선 확인
```

> rebuild는 인덱스를 새로 만들므로 §6의 백업을 먼저 수행한 뒤 실행하십시오.

## 3. 임베딩 교체 (768 → 1024, norm 재측정)

교체 후보는 BGE-M3(1024차원, ctx8192, MIT)입니다. GGUF 모델 다운로드 여부는 **DECISION**입니다. 768 → 1024 차원이 바뀌므로 인덱스는 반드시 전체 재생성하고, 새 모델의 출력 정규화 상태를 다시 측정해야 합니다(nomic은 norm=1.0이지만 BGE-M3가 동일하다는 보장은 없음 → 재측정 필수).

### 3.1 절차

```bash
# 1) (선행) 현재 인덱스 백업 — §6
python app/cli.py rebuild   # 백업 전 현 상태 고정이 필요하면 생략 가능

# 2) LMStudio에 BGE-M3 로드 후, 새 모델로 norm 재측정
#    출력 임베딩의 L2 norm 분포를 확인 (재정규화 여부 결정의 근거)
python scripts/env_check.py   # 차원=1024, norm 통계 출력 확인

# 3) 인덱스 전체 재생성 (차원 768→1024)
python app/cli.py rebuild

# 4) 평가로 교체 효과 정량 증명 (연차휴가 Recall@1=0 → 목표 1)
python eval/evaluate.py
```

### 3.2 norm 재측정 판정

| 측정 결과 | 조치 |
| --- | --- |
| `abs(norm-1.0)<1e-3` (정규화됨) | nomic과 동일. 재정규화 비활성 유지 |
| norm이 1.0이 아님 | 인덱스 빌드 시 L2 정규화 1회 적용하도록 설정 후 재측정 |

> 차원 변경은 인덱스 헤더의 `dim` 필드 불일치를 유발합니다. 헤더 검증 규칙은 [../03_DATABASE/01_인덱스_메타데이터_설계.md](../03_DATABASE/01_인덱스_메타데이터_설계.md) 참조. 임베딩 호출 계약(`/v1/embeddings`, `usage.prompt_tokens=0` 오보고 무시)은 [../04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) 참조.

### 3.3 교체 합격 기준

- 연차휴가 회귀케이스 Recall@1 = 1 (교체 전 0)
- 전체 골든셋 Recall@1/MRR/nDCG가 교체 전 대비 비퇴행
- 변수는 1개씩만 변경(임베딩만 교체)하고 측정 → 개선 귀속 명확화

## 4. 생성 모델 스위칭

생성 3종 모두 로컬, `parallel=4`로 서빙됩니다.

| 모델 | 규모 | ctx | 비고 |
| --- | --- | --- | --- |
| `qwen/qwen3.6-35b-a3b` | 35B MoE(실측) | 8192 | 기본 후보 |
| `qwen/qwen3.5-9b` | 9B급 [추정] | 8192 | 경량 후보 |
| `google/gemma-4-e4b` | 4B급 [추정] | 65536 | 장문 ctx, OOM 주의(§7) |

스위칭 절차:

```bash
# 1) LMStudio에서 대상 생성 모델 로드 (CLI는 모델명만 가리킴)
# 2) reasoning 폴백 동작 확인 — 한 문장 질문으로 스모크
python app/cli.py ask "연차 휴가 신청 절차"
# 3) content 빈값 → reasoning_content 폴백 / <think> 정제 / length 재시도 동작 확인
```

스위칭 시 필수 점검:

- `message.content` 빈값 → `message.reasoning_content` 폴백이 동작하는가
- `<think>` 태그 정제가 적용되는가
- `finish_reason=length`면 `max_tokens*3`로 1회 재시도, 2회째도 length면 경고 부착되는가

> reasoning 폴백 게이트의 계약/구현 상세는 [../04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md). 근본 대응(LMStudio 프리셋 reasoning 종료토큰 점검)의 메뉴 위치는 [확인필요].

## 5. 리랭커 도입 (L3)

리랭커 후보는 BGE-reranker-v2-m3입니다. 검색 1차 결과(하이브리드 RRF 상위 k)를 재정렬해 Recall@1 개선을 노립니다. 로컬 서빙 방식([확인필요], LMStudio가 reranker 엔드포인트를 제공하는지 미확인 → 별도 프로세스/라이브러리 서빙 여부 **DECISION**)을 먼저 확정해야 합니다.

도입 절차(L3 단계에서만):

```bash
# 1) 리랭커 로컬 서빙 기동 (방식 DECISION 후)
# 2) 검색 파이프라인에 재정렬 단계 추가 (변수 1개만 추가)
# 3) 평가로 효과 증명 — 리랭커 on/off 비교
python eval/evaluate.py
```

- 합격 기준: 연차휴가 Recall@1 = 1, 전체 지표 비퇴행
- 리랭커는 임베딩 교체와 동시에 켜지 말 것(변수 분리). 임베딩 교체 → 측정 → 리랭커 → 측정 순서.

## 6. 백업 (rotation / retention)

대상은 인덱스 디렉토리, 인덱스 헤더, 골든셋(`eval/queries.yaml`), 설정 파일입니다. 원문 문서(`data/`)는 외부 원본이 있으면 선택입니다.

### 6.1 백업 시점

- `rebuild` 실행 직전(필수)
- 임베딩 교체·청킹 파라미터 변경 직전(필수)
- 주간 정기점검 시(§7)

### 6.2 rotation / retention 정책

| 항목 | 정책 |
| --- | --- |
| 명명 | `backup/index_YYYYMMDD_HHMMSS/` |
| rotation | 최신 7개 일일 백업 보관 |
| retention | 주간 백업 4개(약 1개월), 월간 백업 3개(약 분기) 별도 보관 |
| 검증 | 백업 직후 헤더 `dim`/스키마 버전 읽기 확인 |

> 보관 개수·경로는 운영 정책으로 조정 가능(**DECISION**). 위 수치는 권장 기본값입니다. 복원 시에는 백업 인덱스로 교체 후 `python eval/evaluate.py`로 동등성을 검증하십시오.

## 7. 정기 점검 (주간)

주 1회 다음 체크리스트를 수행합니다.

```bash
# 1) 환경 점검 (LMStudio 가동/모델 로드/차원/norm)
python scripts/env_check.py

# 2) 골든셋 회귀 평가 (연차휴가 회귀케이스 포함)
python eval/evaluate.py

# 3) 검색 스모크 + 생성 폴백 스모크
python app/cli.py search "연차 휴가 신청 절차"
python app/cli.py ask "연차 휴가 신청 절차"
```

점검 합격 기준:

- `env_check` 통과(차원=현 인덱스 `dim`, norm 가드 통과)
- 골든셋 지표가 직전 측정 대비 비퇴행(특히 연차휴가 Recall@1 유지)
- 생성 폴백 정상(빈 content → reasoning_content, length 재시도)

> 평가 절차의 표준 정의는 [../../시작프롬프트.md](../../시작프롬프트.md) §B-8로 단일화합니다. 본 런북은 실행 시점·주기만 규정하고 지표 정의는 중복하지 않습니다.

## 8. 장애 대응

| 증상 | 추정 원인 | 조치 |
| --- | --- | --- |
| 생성 응답이 빈 문자열 | `message.content` 빈값(3종 공통 BLOCKING) | `reasoning_content` 폴백 동작 확인 → [../04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) |
| 답변 중간 잘림(12,972자 등) | 무한사고로 `finish_reason=length` | `max_tokens*3` 1회 재시도, 2회째도 length면 경고 부착 |
| `gemma-4-e4b` 로드/추론 중 OOM | ctx=65536 + `parallel=4` 동시 점유로 메모리 초과 | ctx 축소 또는 `parallel` 하향, 혹은 모델 단독 로드 후 재시도 |
| 검색 결과 norm 가드 실패 | 임베딩 출력 norm ≠ 1.0 / 재정규화 중복 | §3.2 재측정, 재정규화 설정 점검 |
| 인덱스 차원 불일치 | 임베딩 교체 후 인덱스 미재생성(768 vs 1024) | `python app/cli.py rebuild` 후 평가 |
| 헤더 스키마 버전 불일치 | 인덱스 포맷 변경 | rebuild 또는 백업 복원 → [../03_DATABASE/01_인덱스_메타데이터_설계.md](../03_DATABASE/01_인덱스_메타데이터_설계.md) |
| 연차휴가 질의 오답(Recall@1=0) | 임베딩 한국어 약점(정답 3위, 격차 -0.018) | 임베딩 교체(§3)/리랭커(§5) 후 재측정 |
| `usage.prompt_tokens=0` 보고 | 임베딩 토큰 수 오보고(알려진 현상) | 무시(정상). 토큰 기반 과금/추정 금지 |
| LMStudio 무응답 | 서버 미가동/모델 미로드 | `python scripts/env_check.py`로 원인 격리 → [../01_시스템구성/02_로컬환경_LMStudio실측.md](../01_시스템구성/02_로컬환경_LMStudio실측.md) |

### 8.1 gemma OOM 상세

`google/gemma-4-e4b`는 `ctx=65536`로 가장 큰 컨텍스트를 갖습니다. `parallel=4` 동시 서빙과 결합되면 KV 캐시 메모리가 ctx×병렬수로 증가해 OOM이 발생할 수 있습니다.

대응 우선순위:

1. `parallel` 값을 4 → 1~2로 하향(동시성 ↓)
2. 해당 모델 사용 시 ctx를 실제 필요 길이로 축소
3. 다른 모델 언로드 후 단독 로드
4. 위 조정의 구체적 LMStudio 설정 메뉴 위치는 [확인필요]

## 9. 변경 시 공통 원칙

- 변수는 한 번에 하나만 변경하고 `python eval/evaluate.py`로 정량 증명합니다(임베딩·리랭커·청킹·모델 동시 변경 금지).
- 인덱스를 바꾸는 모든 작업(`rebuild`, 임베딩 교체, 청킹 변경)은 §6 백업을 선행합니다.
- 개선·퇴행 판정은 주관이 아니라 골든셋 지표로만 합니다. 회귀케이스(연차휴가)는 항상 포함합니다.
- 클라우드 API·재정규화·허구 모듈 호출은 어떤 절차에서도 사용하지 않습니다.
