# 프로젝트 개요 (docs-em)

docs-em는 로컬 LMStudio만으로 한국어 사내 문서 검색 품질을 끌어올리는 RAG 파이프라인 프로젝트입니다. 이 문서는 가장 먼저 읽는 **진입점**으로, 무엇을·왜·어떻게의 큰 그림과 전체 문서맵을 제공합니다.

> 관련: [01_기술스택결정서.md](./01_기술스택결정서.md) · [02_로컬환경_LMStudio실측.md](./02_로컬환경_LMStudio실측.md) · [03_개발환경셋업.md](./03_개발환경셋업.md) · 불변식 정본: [99_AI참조/01_실측제약_불변식.md](../99_AI참조/01_실측제약_불변식.md) · 정본 SSOT: [시작프롬프트.md](../../시작프롬프트.md)

---

## 1. docs-em이 무엇인가

docs-em는 **PDF/DOCX 사내 문서를 색인하고 한국어 자연어 질의로 정확히 검색**하는 것을 목표로 하는 RAG(Retrieval-Augmented Generation) 시스템입니다. 핵심 차별점은 **완전 로컬 전용**입니다. 모든 임베딩·생성·리랭킹이 로컬 LMStudio 서버에서 수행되며, 외부 클라우드 API는 사용하지 않습니다.

| 항목 | 내용 |
| --- | --- |
| 서빙 엔드포인트 | `http://localhost:1234/v1` (OpenAI 호환) |
| `api_key` | 더미 문자열 `"lm-studio"` (로컬이라 인증 무의미, 단 비우면 일부 SDK가 거부하므로 더미 필수) |
| 외부 클라우드 API | OpenAI / Anthropic / Cohere 등 **절대 금지** |
| 1순위 목표 | 한국어 문서 검색 품질 |
| 현재 단계 | 코드 없음, 스택 미확정, `docs/`만 작성 중 (구현은 나중) |

> 클라이언트 초기화 예: `OpenAI(base_url="http://localhost:1234/v1", api_key="lm-studio")`. 자세한 실측·진단은 [02_로컬환경_LMStudio실측.md](./02_로컬환경_LMStudio실측.md)(시작프롬프트.md §B-12) 참조.

## 2. 왜 만드는가 (문제 정의)

현재 로컬 임베딩 모델의 **한국어 검색 품질이 실측상 불충분**합니다. 대표 회귀 케이스는 다음과 같습니다.

- 질의: **"연차 휴가 신청 절차"**
- 정답 문서(휴가규정)가 **3위**에 랭크됨
- 오답 문서(출장경비)보다 점수 격차 **-0.018** → **Recall@1 = 0**

즉, 사람이 보기에 명백한 정답을 1위로 올리지 못하는 상태입니다. docs-em의 존재 이유는 이 한국어 검색 약점을 정량 평가로 드러내고, 변수를 하나씩 바꿔가며 개선하는 것입니다.

→ 임베딩 모델별 실측 비교·교체 근거는 [01_기술스택결정서.md](./01_기술스택결정서.md)(시작프롬프트.md §B-5)와 아래 §4 참조.

## 3. 목표와 핵심 제약

### 목표
1. **한국어 문서 검색 품질 (1순위)** — 위 회귀 케이스를 포함한 골든셋에서 Recall@k / MRR / nDCG 개선.
2. 재현 가능한 평가 — 변수를 1개씩만 변경하여 효과를 측정(아래 §7).

### 핵심 제약 (불변식)
아래는 추측이 아니라 **이미 재현된 실측 사실**입니다. 설계·코드에서 반드시 반영하며, 정본은 [99_AI참조/01_실측제약_불변식.md](../99_AI참조/01_실측제약_불변식.md)(INV-1~INV-8, AI 자가점검 체크리스트)입니다.

- **로컬 전용 (INV-3)**: 모든 추론은 `localhost:1234`. 클라우드 API 호출 코드는 작성하지 않습니다.
- **재정규화 금지 (INV-1)**: 현재 임베딩 출력은 이미 L2 정규화되어 있습니다(norm=1.0, 아래 §4). 정규화 벡터끼리는 **코사인 유사도 = 내적**이지만, 이 등가성은 **norm=1.0 전제에서만** 성립합니다. 따라서 **모델 교체(예: BGE-M3) 시 norm을 반드시 재측정**하고 정규화 분기를 갱신해야 합니다(§4·시작프롬프트.md §A-2 ①).
- **토큰 카운트 불신 (INV-4)**: `usage.prompt_tokens=0` 오보고 → 청크 길이는 문자 수 기준 1차, 무음 잘림(silent truncation) 방지 가드 필요(§4·시작프롬프트.md §A-2 ②).
- **reasoning 폴백 필수 (INV-2/5/6)**: 생성모델이 `message.content`를 비우는 BLOCKING 이슈가 있어, 폴백 처리 없이는 답을 받을 수 없습니다(아래 §5).

## 4. 임베딩 현황 (실측)

| 항목 | 실측값 |
| --- | --- |
| 모델 | `text-embedding-nomic-embed-text-v1.5` |
| 차원 | 768 |
| 정규화 | 출력 이미 L2 정규화됨 (norm = 1.0) → **재정규화 금지** (INV-1) |
| prefix 효과 | `search_document` / `query` prefix 효과 미미 (0.687 → 0.695) (INV-8) |
| 토큰 계측 | `usage.prompt_tokens = 0` 오보고 → 토큰 기준 신뢰 불가 (INV-4) |
| 입력 한계 | 임베딩 입력 **최대 길이 / 배치 한계 [확인필요]** — `prompt_tokens=0`이라 응답으로 초과 여부를 알 수 없어 무음 잘림 위험. 로컬 토큰/문자 가드로 방어 |

### 교체 후보 (벤더/벤치 주장 — 미채택 단계)

아래 BGE-M3 수치는 **교체 후보의 공개 스펙·벤치 주장**입니다(임베딩 교체는 아직 수행하지 않은 단계). SSOT는 이를 한국어 우수 후보로 평가하나(시작프롬프트.md §B-5), 채택 전에는 골든셋 실측으로 직접 검증해야 합니다. e5 / KoSimCSE / Solar 등 다른 한국어 후보의 성능은 SSOT에서 [추정]으로 구분되므로, 단일안이 아니라 후보군 중 하나로 봅니다.

| 항목 | BGE-M3 |
| --- | --- |
| 차원 | 1024 (768→1024로 인덱스 전체 재생성 필요) |
| 컨텍스트 | 8192 |
| 검색 방식 | Dense + Sparse + ColBERT |
| 라이선스 | MIT |
| 한국어 | Recall@1 우수 (벤더/벤치 주장 — 골든셋 실측 검증 대상) |
| 배포 | GGUF 1~2GB |
| norm | **재측정 필수** — norm=1.0 보장 안 됨, 교체 시 INV-1 가드 갱신 |

> **DECISION** (시작프롬프트.md §B-9 #2) — BGE-M3 GGUF(1~2GB) 다운로드 및 채택 여부는 사용자 결정 사항입니다. 디스크/다운로드 비용 vs 한국어 변별력 트레이드오프. 허용 시 768→1024 인덱스 재생성 + norm 재측정이 따릅니다.
> **DECISION** (시작프롬프트.md §B-9 #1) — 주 사용 언어가 한국어인가? (예 가정) → 예이면 BGE-M3 교체 강력 권장.

→ 임베딩 교체 확정·실험은 [01_기술스택결정서.md](./01_기술스택결정서.md)(시작프롬프트.md §B-5)에서 진행합니다.

## 5. 생성모델 현황 및 BLOCKING 이슈

로컬 생성모델 3종(모두 로컬, `parallel=4`):

| 모델 | 규모 | 컨텍스트(ctx) | 비고 |
| --- | --- | --- | --- |
| `qwen/qwen3.6-35b-a3b` | 35B MoE | 8192 | |
| `qwen/qwen3.5-9b` | 9B급 [추정] | 8192 | |
| `google/gemma-4-e4b` | 4B급 [추정] | 65536 | |

> 각주: **ctx 수치(8192 / 8192 / 65536)와 35B MoE는 모두 실측**입니다. [추정]은 **파라미터 수(9B급 / 4B급)에만** 걸립니다(시작프롬프트.md §B-3·§B-12, 형제 문서 99_AI참조 표와 동일 처리).

### BLOCKING: content 빈값 이슈 (INV-2/5/6, 시작프롬프트.md §B-4)
세 모델 **모두** `message.content`가 **빈 문자열**로 오고, 실제 답은 `message.reasoning_content`로 옵니다.

- `/no_think`, `enable_thinking=false`, `reasoning_effort=low`, `max_tokens=4000` **전부 무효**
- 무한 사고로 `finish_reason=length` 잘림 (한 문장 질문에 12,972자 생성)

**필수 코드 대응 (3단계)**:
1. `content` 빈값 시 → `reasoning_content` 폴백 (INV-2)
2. `<think>…</think>` 태그 정제·최종답 추출 (INV-5)
3. `finish_reason=length`면 재시도 (INV-6)

**근본 대응 (병행)**: LMStudio **프리셋의 reasoning 종료 토큰(stop 토큰) 점검** — 모델 프리셋에서 종료 토큰이 누락되면 무한 사고가 발생합니다. 해당 설정 **메뉴 위치는 [확인필요]**. 코드 폴백은 증상 완화이고, 프리셋 점검이 근본 경로입니다.

> **DECISION** (시작프롬프트.md §B-9 #8) — 답변 생성 **기본 모델**: 권장값 `gemma-4-e4b`(ctx 65536, 긴 컨텍스트 강점). 단 4B급이라 추론 품질 한계 가능 → 위 BLOCKING 해결 후 재평가. 사용자 결정 사항.

→ 폴백 게이트 구현 명세·코드는 [04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md)와 [99_AI참조/01_실측제약_불변식.md](../99_AI참조/01_실측제약_불변식.md)(시작프롬프트.md §B-4·§B-7) 참조.

## 7. 평가 방법

- **지표**: Recall@k (1/3/5/10), MRR, nDCG
- **골든셋**: 연차휴가 회귀 케이스 포함 (위 §2)
- **원칙**: 변수를 1개씩만 변경하여 측정 (효과 격리). 개선은 체감이 아니라 정량 지표로 증명 (INV-7)

→ 평가 하네스·운영 상세는 [05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md)(시작프롬프트.md §B-8) 참조.
