## 0. 결정 원칙 (Decision Drivers)

이 문서의 모든 선택은 아래 4개 제약을 우선순위 순으로 따릅니다. 실측 불변식의 정본은 → 시작프롬프트.md **§A-2**(반드시 지켜야 할 실측 제약)와 **§B-12**(실측 사실 요약, 변경 금지)이며, INV 체크리스트는 [99_AI참조/01_실측제약_불변식.md](../99_AI참조/01_실측제약_불변식.md) 참조. (§B-2는 데이터 파이프라인 섹션으로 불변식 출처가 아닙니다.)

| # | 제약 | 출처 | 영향 |
|---|------|------|------|
| D1 | **완전 로컬**: 외부 클라우드 API(OpenAI/Anthropic/Cohere) 절대 금지. 추론은 LMStudio(`http://localhost:1234/v1`, OpenAI 호환, `api_key` 더미) 단일 백엔드. | §A-2 · §B-12 (INV-3) | 모든 SDK/모델/리랭커는 로컬 서빙 가능해야 채택 |
| D2 | **한국어 검색 품질 1순위**: 한국어 Recall@1 실측이 채택의 1차 근거 | §A-2 · §B-12 | 외부 사양만으로는 채택 불가, 골든셋 단독 변수 재측정 게이트 필수 |
| D3 | **변수 1개씩 변경·측정 가능**: 계층을 교체 가능한 얇은 인터페이스로 분리 | §A-2 · §B-8 (평가전략) | 추상화 가능한 라이브러리 우선 |
| D4 | **현재 코드 없음·구현은 나중**: 본 문서는 스택을 *고정*하되, 다운로드/서빙 등 비용 발생 항목은 사용자 결정(**DECISION**)으로 남김 | §A-2 · §B-9 | 미확정 항목은 권장값 + DECISION 태그 |

상태 표기 규약(문서 전반 통일):

- **실측 기준선**: 현재 서빙 중인 검증된 값(변경 금지)
- **권장 확정**: 본 문서가 1순위로 고정. 추가 비용·사용자 결정 불필요
- **[확인필요]**: 사실/지원 여부가 미검증인 항목(검증 후 확정)
- **DECISION**: 다운로드·서빙 비용이 발생해 사용자 최종 결정이 필요한 항목

## 2. ADR 본문

각 ADR은 **Context → Options → Decision → Consequences** 형식입니다.

### ADR-01. 언어: Python 3.11+

- **Context**: RAG 생태계(임베딩·벡터DB·형태소·리랭커 바인딩)가 Python 중심. 한국어 NLP 핵심 도구(`kss`, `kiwipiepy`)도 Python.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **Python 3.11+** | 생태계 최대, 3.11 성능 개선·`tomllib` 내장 | GIL |
  | Node/TS | 비동기 강점 | 한국어 형태소·벡터DB 바인딩 빈약 |
  | Rust/Go | 성능 | RAG/NLP 생태계 미성숙, 개발 속도 ↓ |
- **DECISION**: **Python 3.11+** 채택. (권장 확정)
- **Consequences**: 가상환경(uv)/의존성 관리 필요 → 의존성 목록·버전 핀 정책·설치 절차는 [03_개발환경셋업.md](03_개발환경셋업.md) §2 참조(본 문서를 역참조함). 3.11 미만 금지(`tomllib`·예외그룹 등 사용 가능성).

### ADR-02. 추론 SDK: openai SDK (base_url 교체)

- **Context**: LMStudio는 OpenAI 호환 엔드포인트(`/v1/chat/completions`, `/v1/embeddings`) 제공. 클라우드 금지(D1)이므로 SDK는 엔드포인트만 로컬로 돌릴 수 있으면 됩니다.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **`openai` SDK + `base_url`** | 표준 인터페이스, `embeddings`/`chat` 일관, 재시도 내장 | 클라우드 연상(단 호출처는 localhost 고정) |
  | `httpx`/`requests` 직접 | 의존성 최소, 응답 원형 제어 | 재시도·스트리밍 직접 구현 |
- **DECISION**: **`openai` SDK** 채택. 클라이언트 1곳에서 `base_url="http://localhost:1234/v1"`, `api_key="not-needed"`(더미) 고정. (권장 확정)
- **Consequences**:
  - 생성 모델의 **content 빈값·reasoning_content 폴백** 처리가 필수입니다(BLOCKING, INV-2/5/6). 응답 객체에서 `message.content`가 빈문자열이면 `message.reasoning_content`로 폴백 + `<think>` 정제 + `finish_reason=="length"` 재시도. 본 문서는 SDK 선택까지만 다룹니다. → **실행 가능한 정본 코드**는 [시작프롬프트.md](../../시작프롬프트.md) **§B-7**(클라이언트 래퍼), **배경/무효 우회책**은 §B-4, **호출자 함수 계약**은 [04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) §2, **LMStudio 프리셋 reasoning 종료토큰 점검 절차**는 [03_개발환경셋업.md](03_개발환경셋업.md) §0(§B-4 위임) 참조.
  - 임베딩 응답의 `usage.prompt_tokens=0` 오보고는 무시(토큰 회계에 사용 금지, INV-4). 토큰/길이 계산은 ADR-02a 참조.

### ADR-02a. 토큰/길이 계산: 문자 수 1차 + 모델별 HF 토크나이저

- **Context**: 컨텍스트 예산 관리(청크 길이, ctx8192/65536 한계 점검)에 토큰 수가 필요하나, LMStudio가 `usage.prompt_tokens=0`으로 오보고하므로 응답 usage를 신뢰할 수 없습니다(INV-4). 길이는 로컬에서 직접 계산해야 합니다.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **문자 수 1차 + 모델별 HF 토크나이저** | usage 비의존, 모델별 정확, 로컬 | 모델별 토크나이저 로드 필요 |
  | `tiktoken` 단독 | 빠름, 의존성 단순 | OpenAI BPE라 로컬 모델 ctx 예산엔 부정확 |
- **DECISION**: **문자 수 기준 1차 제어 + 모델별 HF 토크나이저로 정밀 토큰 카운트** 채택. `tiktoken`은 토크나이저 미가용 시 **근사 폴백**으로만 사용. (권장 확정)
- **Consequences**: 모든 길이/예산 판단은 로컬 계산을 1차 근거로 사용. 응답 `usage`는 회계에 쓰지 않습니다. 청크 길이 제어(ADR-03a)도 동일 계산 계층을 공유합니다.

### ADR-03. 문서 파싱: pymupdf + python-docx

- **Context**: 사내 문서 입력은 PDF·DOCX 중심으로 가정. 한국어 텍스트 추출 정확도와 레이아웃(표·헤더) 보존이 청킹 품질을 좌우합니다.
- **Options**:
  | 옵션 | 대상 | 장점 | 단점 |
  |------|------|------|------|
  | **`pymupdf`(fitz)** | PDF | 빠름, 블록/스팬 좌표 제공(레이아웃 청킹 가능), 순수 로컬 | 표 구조 복원은 추가 작업 |
  | **`python-docx`** | DOCX | 단락/스타일/표 구조 직접 접근 | 텍스트 외 렌더링은 부재 |
  | `unstructured` | 다포맷 | 일괄 파이프라인 | 의존성 무거움, 내부 외부 호출 옵션 주의(로컬 한정 필요) |
  | `markitdown` | 다포맷 | 마크다운 변환 편의 | 한국어 표/레이아웃 정밀도 [확인필요] |
  | `pdfplumber` | PDF | 표 추출 우수 | 대용량에서 느림 |
- **DECISION**: **PDF=`pymupdf`, DOCX=`python-docx`** 채택. 표 추출이 핵심인 문서가 다수로 확인되면 `pdfplumber` 보조 도입 검토 [확인필요]. (권장 확정)
- **Consequences**: 포맷별 추출기를 공통 `Document(text, meta)` 인터페이스로 감싸 D3(교체 가능성) 충족. 스캔 PDF(OCR 필요) 처리 범위는 본 문서 밖.

### ADR-03a. 청킹 분할기: langchain-text-splitters (분할기만)

- **Context**: 파싱된 텍스트를 인덱싱 단위로 쪼개는 **분할기(splitter)** 자체의 결정입니다. 한국어 문장 경계(ADR-04)·형태소(ADR-05)는 별개이며, 여기서는 문자 기준 길이 제어와 구분자 기반 재귀 분할 도구만 다룹니다. (청킹 **전략/파라미터**는 비범위 — §4 참조.)
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **`langchain-text-splitters`(분할기 모듈만)** | 재귀 분할·문자 길이 제어 검증됨, LangChain 본체 불필요 | 외부 의존성 1개 추가 |
  | 자체 구현 | 의존성 0, 완전 제어 | 경계 처리·overlap 직접 구현 |
- **DECISION**: **`langchain-text-splitters`(분할기 컴포넌트만)** 채택. 문자 수 기준 길이 제어(ADR-02a 계산 계층 공유)와 한국어 문장 경계(ADR-04 `kss`)를 구분자로 결합. 의존성 최소화가 더 중요해지면 자체 구현으로 강등 가능. (권장 확정)
- **Consequences**: LangChain **오케스트레이션 본체는 채택하지 않습니다**(ADR-12, 분할기 부품만 빌려 씀). 청크 크기/overlap/구조 기반 전략과 청크 데이터모델·내부 `chunk()` 함수 계약은 → [04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) §3.1 및 시작프롬프트.md **§B-2 ②**(청킹—한국어 특화) 참조.

### ADR-04. 문장 분리: kss

- **Context**: 청킹 전 한국어 문장 경계 분리가 필요. 종결어미·인용·약물 등 한국어 특수 패턴 처리 품질이 관건.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **`kss`** | 한국어 전용 문장분리, 규칙·예외 정교 | 매우 긴 입력 속도 [확인필요] |
  | `kiwipiepy` `split_into_sents` | 형태소 분석기와 도구 일원화 | 분리 규칙 세밀도는 kss 대비 비교 필요 |
  | 정규식 split | 의존성 0 | 한국어 종결 패턴 오분리 다발 |
- **DECISION**: **`kss`** 채택(문장 분리). 형태소/토크나이저는 별도(ADR-05). (권장 확정)
- **Consequences**: `kss`(문장)와 `Kiwi`(형태소) 2개 한국어 도구를 함께 사용. 일원화(Kiwi 단일)는 골든셋 비교 후 재검토 가능 [확인필요].

### ADR-05. 형태소 분석 / BM25 토크나이저: Kiwi (kiwipiepy)

- **Context**: BM25(ADR-08)는 한국어에서 **형태소 토크나이즈**가 품질을 좌우합니다. 공백 분리만으로는 조사·어미 변형 때문에 매칭이 깨집니다.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **`Kiwi`(kiwipiepy)** | 순수 Python 설치 간편(C++ 백엔드 내장), 빠름, 사용자 사전, 문장분리 겸용 가능 | — |
  | `mecab-ko` | 성숙·정확 | 시스템 의존성(사전 설치) 번거로움 |
  | `Okt`(konlpy) | konlpy 통합 | JVM 의존, 속도 |
  | `soynlp` | 비지도 학습 | 도메인 사전 구축 비용 |
- **DECISION**: **`Kiwi`** 채택(형태소·BM25 토큰화). 명사/어간 중심 토큰 추출로 BM25 인덱싱. (권장 확정)
- **Consequences**: 사용자 사전(사내 용어·약어)으로 검색 품질 개선 여지. 동일 토크나이저를 인덱싱·질의 양쪽에 일관 적용해야 함(불일치 시 매칭 붕괴).

### ADR-06. 임베딩 (L1 기준선): nomic-embed-text-v1.5

- **Context**: 현재 LMStudio에 서빙 중인 임베딩 모델이며, L1 검색 MVP의 **측정 기준선**입니다.
- **실측 사실** (§A-2 · §B-12, 변경 금지):
  | 항목 | 값 |
  |------|-----|
  | 모델 | `text-embedding-nomic-embed-text-v1.5` |
  | 차원 | 768 |
  | 정규화 | 출력 이미 **L2 정규화됨(norm=1.0)** → **재정규화 금지**(INV-1) |
  | prefix 효과 | `search_document`/`query` prefix 효과 **미미** (0.687 → 0.695) |
  | 토큰 보고 | `usage.prompt_tokens=0` **오보고**(무시, INV-4) |
  | 한국어 약점 | "연차 휴가 신청 절차" 질의에 정답(휴가규정) **3위**, 오답(출장경비)보다 격차 **-0.018** → **Recall@1=0** |
- **DECISION**: L1에서는 **본 모델 유지**(기준선 측정). 정규화 코드 추가 금지, prefix 적용은 선택(효과 미미). (실측 기준선)
- **Consequences**: 위 한국어 약점이 L2 임베딩 교체(ADR-07)의 트리거. "연차휴가" 케이스는 골든셋 **회귀 케이스**로 고정 → 평가 사양은 시작프롬프트.md **§B-8**(Recall@k 골든셋) 및 [05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md) §2(교체 후 재생성·재측정) 참조.

### ADR-07. 임베딩 (L2 교체후보): BGE-M3

- **Context**: ADR-06의 한국어 Recall@1=0 약점을 해소하기 위한 1순위 교체후보.
- **후보 사양** (BGE-M3, **외부 사양 인용** — 본 프로젝트 골든셋 실측 아님):
  | 항목 | 값 |
  |------|-----|
  | 차원 | 1024 |
  | 컨텍스트 | 8192 |
  | 기능 | Dense + Sparse + ColBERT(multi-vector) |
  | 라이선스 | MIT |
  | 한국어 | Recall@1 우수 — **외부 사양 인용**(본 프로젝트 측정값 아님) |
  | 배포 | GGUF 1~2GB |
- **Options**: BGE-M3가 1순위. 타 한국어 임베딩(예: KURE 계열 등)은 BGE-M3 대비 골든셋 비교가 전제 [확인필요].
- **DECISION (사용자 결정 필요)**: **BGE-M3 GGUF 다운로드(1~2GB) 및 LMStudio 서빙 여부는 DECISION 항목**. 다운로드 비용은 사용자가 결정합니다. (DECISION)
  - **채택 게이트(D2 필수)**: BGE-M3의 한국어 Recall@1 우수는 외부 사양일 뿐이며, **본 프로젝트에서 측정된 값은 nomic의 -0.018 / Recall@1=0 뿐**입니다. 따라서 채택 전 **"연차휴가" 회귀 케이스를 포함한 골든셋에서 임베딩 변수만 단독 변경해 Recall@1을 직접 재측정**(D2/D3)하고, nomic 대비 개선이 숫자로 확인될 때만 승격합니다.
  - 채택 시: 차원이 768 → **1024로 변경**되므로 벡터스토어 인덱스 **재생성** 필요(ADR-09 참조).
  - BGE-M3의 Sparse/ColBERT 활용 범위는 L2/L3 하이브리드 설계와 연동 [확인필요].
- **Consequences**: 교체는 D3에 따라 **임베딩 변수만 단독 변경**해 골든셋 재측정. 교체 시 **norm 재측정 필수**(ADR-09 메트릭 경고·INV-1 참조).

### ADR-08. BM25 (희소 검색): rank_bm25

- **Context**: Dense 임베딩 약점(특히 ADR-06 한국어 케이스)을 보완하는 어휘 매칭(lexical) 채널.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **`rank_bm25`** | 순수 Python, 의존성 가벼움, in-memory 빠른 프로토타입 | 대규모 영속화·증분색인 약함 |
  | Qdrant/OpenSearch sparse | 확장·통합 하이브리드 | 인프라 무거움(로컬 단일 노드엔 과함) |
  | BGE-M3 Sparse | 임베딩과 단일 모델 | ADR-07 채택·서빙 선행 필요 |
- **DECISION**: **`rank_bm25`** 채택. 토크나이저는 **Kiwi**(ADR-05). BGE-M3 Sparse는 ADR-07 채택 후 대안으로 비교 [확인필요]. (권장 확정)
- **Consequences**: 인덱싱/질의에 **동일 Kiwi 토큰화** 강제. 코퍼스 증가 시 영속화 전략 재검토.

### ADR-09. 벡터스토어: LanceDB 또는 ChromaDB

- **Context**: Dense 벡터 저장·ANN 검색·메타데이터 필터가 필요. 완전 로컬·임베디드(서버 불필요) 우선.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **LanceDB** | 임베디드, 컬럼형(Lance)·디스크 효율, 메타필터, 빠른 성장 | 생태계 비교적 신생 |
  | **ChromaDB** | 임베디드, API 단순, RAG 예제 풍부 | 대규모 성능 [확인필요] |
  | Qdrant | 하이브리드·필터 고급 | 서버 구동(로컬 단일 노드엔 과할 수 있음) |
  | FAISS(raw) | 최고속 ANN | 메타데이터·영속화 직접 구현 |
- **DECISION (사용자 결정 필요)**: **LanceDB 또는 ChromaDB 중 택1은 DECISION 항목**. 둘 다 임베디드·로컬·D1 충족. (DECISION) 권장 기준:
  - 디스크 효율·컬럼형 메타필터 중시 → **LanceDB**
  - 최소 구성·예제 친화 → **ChromaDB**
- **Consequences**:
  - **거리 메트릭**: 출력이 L2 정규화됨(ADR-06, norm=1.0)이므로 **코사인 = 내적(dot) 등가** → cosine/IP 중 선택. **⚠️ 단, 이 등가성은 `norm=1.0` 전제(nomic)에서만 성립합니다.** BGE-M3(1024d) 등으로 교체(ADR-07)하면 norm이 보장되지 않을 수 있으므로 **교체 시 반드시 norm을 재측정**하고, 정규화가 깨졌으면 정규화 분기를 갱신하거나 metric을 명시적으로 `cosine`(스토어 내부 정규화 의존)으로 고정해야 합니다(INV-1). 메트릭을 무조건 cosine=IP로 고정하면 안 됩니다.
  - **임베딩 차원 변경(768→1024, ADR-07) 시 인덱스 재생성 필요**. 재생성을 자동 무효화하는 **구조**(인덱스 헤더 `embedding_model`/`dim`/`normalized`/`metric` 슬롯, 768→1024 스키마 차이)와 **절차**는 → [03_DATABASE/00_벡터스토어_스키마.md](../03_DATABASE/00_벡터스토어_스키마.md) §1-1(인덱스 헤더)·§2(메트릭)·§7(768→1024 스키마 차이) 및 [05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md) §2(전체 재생성 절차) 참조. *(`03_DATABASE/01_인덱스_메타데이터_설계.md`는 미작성(TBD) — 작성 시 본 항목과 상호 링크.)*
  - 벡터스토어는 얇은 `VectorStore` 인터페이스 뒤에 두어 양자 교체 비용 최소화(D3).

### ADR-10. 하이브리드 결합: RRF (Reciprocal Rank Fusion)

- **Context**: Dense(임베딩) + Sparse(BM25) 두 채널 결과를 하나로 결합. 두 채널의 점수 스케일이 다름(코사인 vs BM25 raw).
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **RRF** | 점수 스케일 무관(순위 기반), 튜닝 파라미터 적음(k), 강건 | 점수 크기 정보 손실 |
  | 가중합(α·dense + β·sparse) | 점수 크기 반영 | 스케일 정규화·α 튜닝 필요, 불안정 |
- **DECISION**: **RRF** 채택. 스케일이 이질적인 두 채널 결합에 강건하고 튜닝 변수가 적어 D3(단일 변수 측정)에 부합. (권장 확정)
- **Consequences**: RRF `k` 상수(SSOT §B-6 기본값 `k=60`)는 골든셋으로 1개 변수씩 조정. RRF 정본 코드는 → 시작프롬프트.md §B-6, 내부 `rrf()` 함수 계약은 [04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) §4.2 참조. BGE-M3 채택 시 Dense/Sparse/ColBERT 3채널 결합으로 확장 가능 [확인필요].

---

### ADR-11. 리랭커 (L3): BGE-reranker-v2-m3

- **Context**: 1차 검색(하이브리드 top-k) 결과를 cross-encoder로 재정렬해 정밀도 향상. L3 단계 항목.
- **Options**: 한국어·다국어 cross-encoder 리랭커 중 **BGE-reranker-v2-m3**가 1순위(BGE-M3 계열 일관성, 다국어 성능). 대안은 LLM 리랭킹(추론 부하 ↑).
- **DECISION (사용자 결정 필요)**: **BGE-reranker-v2-m3** 채택(권장). **리랭커 모델 다운로드도 비용이 발생하므로 BGE-M3와 함께 사용자 결정(DECISION) 항목**이며, **서빙 방식은 [확인필요]**입니다. (DECISION + [확인필요])
- **[확인필요] 서빙 방식**: LMStudio는 chat/embeddings 위주로, **리랭커(cross-encoder rerank) 엔드포인트 지원 여부 확인 필요**. 가능 옵션:
  1. 로컬 별도 프로세스(`sentence-transformers`/`FlagEmbedding`)로 in-process 로드,
  2. 별도 추론 서버.
  → 어느 쪽이든 **D1(완전 로컬) 유지**가 전제. 구체 서빙 결정은 L3 진입 시 확정.
- **Consequences**: 리랭커는 파이프라인 **맨 끝 선택적 단계**로 두어 on/off 변수로 측정(D3). 내부 `rerank()` 함수 계약은 [04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) 참조.

### ADR-12. 오케스트레이션: 자체 얇은 파이프라인

- **Context**: 각 계층을 단일 변수로 측정·교체(D3)하려면 프레임워크의 암묵적 추상화가 오히려 방해가 될 수 있습니다. 또한 reasoning_content 폴백·정규화 스킵 같은 LMStudio 특수 처리를 프레임워크가 가리면 디버깅이 어려워집니다.
- **Options**:
  | 옵션 | 장점 | 단점 |
  |------|------|------|
  | **자체 얇은 파이프라인** | 계층 경계 명확, 변수 1개 교체 용이, 의존성 최소, 디버깅 투명 | 공통 유틸 직접 구현 |
  | LlamaIndex | RAG 특화 | 추상화 두꺼움, 로컬·reasoning폴백 커스터마이즈 마찰 |
  | Haystack | 컴포넌트 풍부 | 동일 추상화 비용, 버전 변동 |
- **DECISION**: **자체 얇은 파이프라인** 채택. 인터페이스 경계: `Parser → Chunker → Embedder → {VectorStore, BM25} → Fusion(RRF) → Reranker → Generator`. 단, **분할기(ADR-03a)·파서(ADR-03) 같은 부품 단위 라이브러리는 빌려 씁니다**(프레임워크 본체만 회피). (권장 확정)
- **Consequences**: 각 노드를 교체 가능한 함수/클래스로 정의 → 골든셋에서 **변수 1개씩** 변경 측정 가능. reasoning_content 폴백(ADR-02) 같은 로컬 특수 처리를 프레임워크 우회 없이 직접 제어. 노드별 함수 계약은 [04_아키텍처_API/02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) 참조.
