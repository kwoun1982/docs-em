## 2. 청크 논리 레코드 스키마

한 개의 청크 = 벡터스토어의 한 개 레코드(=한 개 벡터). 청크와 임베딩 벡터는 **1:1**입니다.

아래 표의 **필드명·필수 여부는 물리 정본** [00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §1과 일치시킵니다. 물리 스키마에 없는 필드(예: 인덱싱 시점에만 쓰고 적재하지 않는 보조 필드)는 "적재" 열에서 명시합니다.

| 필드 | 타입 | 필수 | 적재(물리) | 설명 |
|---|---|:---:|:---:|---|
| `id` (=`chunk_id`) | str | O | O | 청크 전역 고유 ID. 포맷·파싱은 §5. 물리 정본 필드명은 `id`, 인메모리 타입은 `chunk_id` |
| `doc_id` | str | O | O | 원문 문서 ID. 같은 문서의 청크는 동일 `doc_id` 공유 |
| `source_path` | str | O | O | 원문 파일 경로(절대 또는 코퍼스 루트 상대) |
| `chunk_index` | int | O | O | 문서 내 청크 순번(0-base). 같은 `doc_id` 안에서 0,1,2,… |
| `text` | str | O | O | 임베딩·검색 대상 본문(청킹 후 한국어 텍스트) |
| `vector` (=`embedding`) | float32[dim] | O | O | 임베딩 벡터. dim은 **모델 종속**(nomic=768 / BGE-M3=1024). §4 |
| `page` | int \| null | △ | O(메타) | PDF 등 페이지 정보. 없으면 키 생략(§4 주의) |
| `section` | str \| null | △ | O(메타) | 상위 제목/조항 마커(예: "휴가규정 > 제3장 연차"). 추출 가능 시 |
| `char_len` | int | △ | O | 청크 문자 수. **물리 정본 필드명은 `char_len`**(토큰 미신뢰, 문자 기준 길이 제어). §B-2 ② |
| `char_start` | int | △ | ✗(미적재) | 원문 내 시작 문자 오프셋. **인덱싱 시점 보조값**, 물리 스키마에 없음. §2.1 |
| `char_end` | int | △ | ✗(미적재) | 원문 내 끝 문자 오프셋(반열린 구간 `[start, end)`). **인덱싱 시점 보조값**. §2.1 |

> **필드명 정합화 결정**:
> - 길이 필드는 형제 물리 정본을 따라 **`char_len`** 하나로 통일합니다(이전 초안의 `n_chars`는 폐기). `char_len == char_end - char_start`로 산출합니다.
> - ID 필드는 물리 정본이 `id`이고 인메모리 타입([Chunk](../04_아키텍처_API/02_내부API_인터페이스.md) §1)이 `chunk_id`입니다. **둘은 동일 값**이며, 적재 시 `id`, 코드 변수로는 `chunk_id`로 부릅니다.
> - `token_count`/`created_at`은 **물리 정본에 없으므로 적재 필드에서 제거**합니다. 토큰 수는 `usage.prompt_tokens=0` 오보고(§4)로 신뢰 불가하므로 스키마 필드로 두지 않습니다.

### 2.1 `char_start`/`char_end`의 위치 — 적재 필드 아님

`char_start`/`char_end`(원문 오프셋)는 물리 스키마([00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §1)에 **존재하지 않습니다.** 따라서 다음과 같이 처리합니다.

- **기본**: 인덱싱 파이프라인 내부에서만 계산·사용하고 **벡터스토어에는 적재하지 않습니다.** 적재 메타는 `page`/`section`/`char_len`까지입니다.
- **원문 위치 복원이 필요하면**: 인메모리 타입 `Chunk.meta`(dict)에 `char_start`/`char_end`를 담아 흐르게 하거나([02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) §1의 `meta` 필드), 별도 사이드카로 보관합니다. → **DECISION**: 오프셋 적재 여부와 보관 위치(메타 컬럼 추가 vs 사이드카)는 운영 요구로 결정. 적재하기로 하면 [00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §1 표에 필드를 **추가 정의해야** 합니다.

### 2.2 예시 레코드 (논리, JSON)

```json
{
  "id": "sha1:7f3a9c#0",
  "doc_id": "sha1:7f3a9c",
  "source_path": "corpus/hr/휴가규정.md",
  "chunk_index": 0,
  "text": "제3장 연차휴가. 직원은 1년간 80% 이상 출근한 경우 15일의 유급휴가를 받는다. 연차 휴가 신청은 …",
  "vector": ["…768개 float…"],
  "page": null,
  "section": "휴가규정 > 제3장 연차휴가",
  "char_len": 412
}
```

> `vector`는 물리 적재 형태가 스토어 종속입니다. ChromaDB는 `documents`(=`text`)·`metadatas`·`embeddings`를 분리 저장하고, LanceDB는 컬럼형 테이블에 둡니다. 매핑 정본은 [00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §4·§5입니다.

## 3. 한국어 청킹 전략과 메타 필드의 관계

### 3.1 청킹 원칙

1순위 목표가 **한국어 문서 검색 품질**이므로, 청크 경계가 한국어 의미 단위를 깨지 않도록 합니다(시작프롬프트.md §B-2 ②).

| 단계 | 도구(권장) | 산출 | 영향받는 필드 |
|---|---|---|---|
| 문장 분리 | `kss`(또는 종결어미 규칙 기반) | 문장 리스트 | `text` 경계, `char_start/end`(보조값) |
| 형태소·토큰 | `Kiwi` | 토큰(주로 BM25·sparse 검색측) | (적재 필드 없음. BM25 인덱스용) |
| 청크 묶기 | 자체 슬라이딩 윈도우 | 문장 N개 묶음 | `chunk_index`, `char_len` |

> 도구 스택은 **권장·미확정**입니다 → [시작프롬프트.md](../../시작프롬프트.md) §B-6, 결정 포인트 §B-9. `kss`/`Kiwi`는 권장 후보이며 **DECISION** 대상입니다. 토큰 수는 적재하지 않으므로(§2 주의), `Kiwi` 토큰은 BM25 인덱스 구축에만 쓰고 청크 레코드 필드로 저장하지 않습니다.

### 3.2 경계 규칙

- **문장 경계 우선**: 청크는 항상 문장 경계에서 끊습니다(문장 중간 절단 금지). 단순 `.` 분할은 한국어 종결어미·소수점·약어와 충돌하므로 금지합니다(시작프롬프트.md §B-2 ②). `kss`로 분리한 문장 단위를 합쳐 목표 크기에 맞춥니다.
- **의미 단위 우선**: 사규·매뉴얼은 "제N조", "1." 같은 마커가 천연 경계입니다. 구조 마커를 `section`에 누적합니다.
- **목표 크기·오버랩**: 아래 §3.3의 **SSOT 표를 정본으로** 사용합니다. 임베딩 ctx 한계가 아니라 **검색 단위 응집도**를 위한 값이며, 골든셋 평가로 조정합니다(변수 1개씩 변경 → 시작프롬프트.md §B-8).
- **오버랩과 오프셋**: 오버랩이 있어도 보조값 `char_start/end`는 각 청크의 실제 원문 범위를 그대로 기록합니다(구간이 겹칠 수 있음).
- **임베딩 입력 가드**: 청크가 임베딩 모델 입력 한계(Phase 0 실측, nomic-v1.5 한계는 [확인필요])를 넘지 않는지 청킹 단계에서 검사하고, 넘으면 분할합니다. `prompt_tokens=0` 오보고로 모델이 초과를 알려주지 않으므로 이 가드가 없으면 무음 잘림이 발생합니다(시작프롬프트.md §B-2 ②).

### 3.3 청크 크기·오버랩 (SSOT 표 인용 — 문자 기준)

본 문서는 SSOT [시작프롬프트.md](../../시작프롬프트.md) §B-2 ② 표를 **그대로 인용**합니다. 길이 제어는 **문자 수 기준**입니다(토큰 미신뢰, §B-2 ②). `[추정]`은 한국어 일반 권장값으로, 반드시 §B-8 Recall@k로 검증하며 조정합니다.

| 문서 유형 | 청크 크기 | 오버랩 | 근거 |
|---|---|---|---|
| 사규/매뉴얼/FAQ | 300~500자 | 50~100자(10~20%) | 조항 단위가 짧고 독립적 |
| 서술형 보고서/논설 | 500~800자 | 80~150자 | 문맥 연속성 필요 |
| 코드/표 포함 기술문서 | 구조 단위(함수/표) | 0 또는 헤더 반복 | 의미 단위 보존 |

> 위 표가 정본입니다. 이전 초안의 "약 300~600자 / 오버랩 1~2문장"은 SSOT와 단위(문자 vs 문장)·수치가 어긋나므로 **폐기**합니다. 오버랩은 **문자 수/비율** 단위로 구현합니다(문장 단위 아님).

### 3.4 청킹과 한국어 약점의 연결

실측상 "연차 휴가 신청 절차" 질의에서 정답(휴가규정)이 3위, 오답(출장경비)과 격차 −0.018로 **Recall@1=0**입니다(→ 시작프롬프트.md §A-2 ④). 청킹은 이 문제에 다음과 같이 직결됩니다.

- 정답 문장이 **다른 청크로 흩어지면**(예: "연차"와 "신청 절차"가 분리) 단일 청크 점수가 약해집니다 → 의미 단위 청킹·오버랩으로 완화.
- "휴가를/휴가는/휴가의"처럼 조사·어미로 표면형이 분산되면 sparse 검색이 흩어집니다 → `text`를 `Kiwi` 형태소 BM25 신호로 활용해 하이브리드에서 보강(시작프롬프트.md §B-6, [02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md) §4).
- **이 문서 범위 밖**의 임베딩 교체(BGE-M3, §B-5)·리랭킹(§B-2 ⑥)이 근본 해결이며, 청킹은 보조 변수입니다.

| 모델 | `text-embedding-nomic-embed-text-v1.5` | 로컬 LMStudio(`http://localhost:1234/v1`) |
| 차원 | **768** | `vector` 길이는 정확히 768 (INV-1) |
| 정규화 | **이미 L2 정규화됨(norm=1.0)** | **재정규화 금지** (INV-1, §B-7 norm 가드) |
| prefix | search_document/query | 효과 미미(코사인 0.687→0.695 실측). 적용 여부 **DECISION**(INV-8) |
| 토큰 보고 | `usage.prompt_tokens=0` 오보고 | 토큰 수를 스키마 필드로 두지 않음(INV-4) |

- **차원 하드코딩 금지**: 모델 교체(BGE-M3, **768→1024**) 시 `vector` 길이가 바뀌므로, dim은 스키마에 직접 박지 말고 **인덱스 헤더 설정값**(`dim`)으로 관리합니다. 헤더·재빌드 트리거 정본은 [00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §1-1·§7·§8입니다.
- **모델 교체 시 norm 재측정**: BGE-M3는 정규화를 보장하지 않을 수 있어, 교체 시 norm을 반드시 재측정하고 정규화 분기·metric을 갱신합니다(INV-1, 시작프롬프트.md §B-5·§B-7).

> 외부 클라우드 임베딩 API(OpenAI/Anthropic/Cohere 등) 사용 **금지**(INV-3). 모든 임베딩은 로컬 LMStudio에서 생성합니다. OpenAI SDK는 `base_url`만 교체해 사용합니다.

## 5. ID 생성·파싱 규칙

### 5.1 원칙: 결정적(deterministic) ID

같은 입력에 항상 같은 ID가 나오도록 **내용 기반 해시**를 사용합니다. 재색인 시 중복 적재·고아 레코드를 막기 위함입니다.

| ID | 생성식(권장) | 예 |
|---|---|---|
| `doc_id` | `"sha1:" + sha1(정규화된 원문 바이트 또는 source_path)` | `sha1:7f3a9c` |
| `id`(=`chunk_id`) | `doc_id + "#" + str(chunk_index)` | `sha1:7f3a9c#0` |

> **ID 구분자 정합화 결정**: 형제 물리 정본 [00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §1(`{doc_id}#{chunk_index}`)과 통일해 **`#`** 를 구분자로 확정합니다(이전 초안의 `:` 구분자는 폐기). `doc_id`가 `sha1:` 접두로 콜론을 포함하더라도, `#`은 `doc_id`에 등장하지 않으므로 구분자 충돌이 발생하지 않습니다.

### 5.2 파싱 규칙 (역분해)

`chunk_id → (doc_id, chunk_index)` 역분해는 **마지막 `#` 기준 1회 분리**(`rsplit("#", 1)`)로 명문화합니다.

```python
def parse_chunk_id(chunk_id: str) -> tuple[str, int]:
    # 예: "sha1:7f3a9c#0" -> ("sha1:7f3a9c", 0)
    doc_id, idx = chunk_id.rsplit("#", 1)  # 마지막 '#' 기준 분리
    return doc_id, int(idx)
```

- `doc_id`에는 `#`을 절대 사용하지 않습니다(접두는 `algo:hex` 형식의 `:`만 허용). 이 규칙이 깨지면 역분해가 불가능해지므로 ID 생성 시 강제합니다.
- **DECISION**: `doc_id` 입력 — (a) 파일 내용 해시(내용 변경 시 새 ID) vs (b) `source_path` 해시(경로 고정, 내용 변경해도 동일 ID). 재색인 정책에 따라 선택합니다.
- **DECISION**: 해시 알고리즘(sha1 vs sha256). 본 문서는 가독성·충돌 회피를 위해 `algo:hex` 접두 형식을 권장합니다.

### 5.3 멱등성 규칙

- 동일 문서 재청킹 시 같은 `id`가 나와야 하며, **upsert**(있으면 갱신, 없으면 삽입)로 적재합니다.
- 문서가 줄어 청크 수가 감소하면 이전 `chunk_index` 꼬리 레코드가 **고아(stale)**가 됩니다 → 재색인 시 같은 `doc_id`의 청크를 **먼저 삭제 후 재삽입**하는 정책을 권장합니다. 구체 절차·재빌드 트리거는 [00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) §8 정본을 따릅니다.

## 6. 무결성 점검 (인덱싱 시)

검사를 **청크 단독으로 검증 가능한 것**과 **원문 길이를 요구하는 것**으로 분리합니다.

### 6.1 청크 단독 검증 (항상 가능)

| 검사 | 기대 | 위반 시 |
|---|---|---|
| `len(vector) == dim설정값` | 768(또는 교체 후 1024, 인덱스 헤더 `dim`과 일치) | 적재 거부 |
| `char_len == char_end - char_start` | 참(오프셋 보유 시) | 재계산 |
| `0 <= char_start < char_end` | 참(오프셋 보유 시) | 오프셋 로깅·스킵 |
| `chunk_index` 연속(0..K-1) | 참(문서별) | 경고(누락 청크 점검) |
| `id` 전역 유일 | 참 | upsert로 처리 |
| `text` 비어있지 않음 | 참 | 빈 청크 제거 |
| `id == doc_id + "#" + chunk_index` | 참 | ID 재생성 |

### 6.2 원문 길이를 요구하는 검증 (입력 보유 시에만)

`char_end <= len(원문)` 검사는 **원문 전체 문자열 길이를 인덱싱 입력으로 보유할 때만** 수행합니다. 청크 레코드에는 원문 길이 필드가 없으므로, 이 검사는 청킹 함수가 원문을 손에 쥔 **인덱싱 시점**에서만 가능합니다.

| 검사 | 수행 조건 | 위반 시 |
|---|---|---|
| `char_end <= len(원문)` | 청킹 단계에서 원문 문자열 보유 시 | 오프셋 계산 버그 → 적재 거부 |

> 원문(파일) 레벨 스키마 문서는 **현재 미작성**입니다(아래 §8). 원문 보관·길이 확보 책임이 별도 문서로 분리되기 전까지는, 이 검사를 **청킹 함수 내부(원문 인메모리 시점)**에서만 수행하고 적재 후 사후 검증 대상으로 두지 않습니다.
