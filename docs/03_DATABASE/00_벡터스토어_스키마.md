## 1. 논리 스키마 (스토어 무관)

벡터스토어는 **청크 1건 = 레코드 1건**으로 저장합니다. 기본키는 `chunk_id`로 통일합니다(내부 API의 `Chunk.chunk_id`와 동일 필드명 — 형제 문서 [02_데이터모델_청크.md](./02_데이터모델_청크.md)·[내부 API](../04_아키텍처_API/02_내부API_인터페이스.md)와 합의).

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `chunk_id` | string (PK) | 청크 고유 식별자. 권장 형식 `{doc_id}#{chunk_index}` |
| `doc_id` | string | 원문 문서 식별자 |
| `source_path` | string | 원문 파일 경로 |
| `chunk_index` | int | 문서 내 청크 순번 |
| `text` | string | 청크 본문 (검색·생성 컨텍스트) |
| `vector` | float[768] | 임베딩 벡터. 현재 768차원, 출력 이미 L2 정규화됨(norm=1.0) |
| `page` | int? | 원문 위치(페이지). PDF 등에서 사용 |
| `section` | string? | 원문 위치(섹션/조항). 사규·매뉴얼에서 사용 |

> `vector`의 출처는 내부 API `embed()`입니다(→ [내부 API](../04_아키텍처_API/02_내부API_인터페이스.md)). 임베딩 출력은 **이미 정규화**되어 있으므로 저장 전 재정규화하지 않습니다(INV-1, [시작프롬프트.md §A-2 ①](../../시작프롬프트.md)).

### 1.1 `meta` dict ↔ 평면 필드 매핑

내부 API는 위치 정보를 `Chunk.meta: dict`로 묶어 전달하지만, 벡터스토어는 필터링·정렬을 위해 **평면 컬럼**으로 저장합니다. 변환 규칙은 다음과 같습니다.

| 내부 API (`meta` dict 키) | 벡터스토어 평면 컬럼 | 비고 |
| --- | --- | --- |
| `meta["page"]` | `page` | 없으면 NULL |
| `meta["section"]` | `section` | 없으면 NULL |
| (그 외 키) | `meta_json` (직렬화 문자열) | 스키마 외 메타는 JSON으로 보존 |

## 2. 거리 metric 규약 (중요)

현재 임베딩(`text-embedding-nomic-embed-text-v1.5`)은 출력이 이미 L2 정규화되어 있어 `norm=1.0`입니다. 정규화된 벡터끼리는 **코사인 유사도 = 내적(dot)** 이 성립하므로 `cosine`과 `dot` 중 무엇을 써도 순위가 동일합니다.

> **⚠️ metric 기본값 경고**: LanceDB의 거리 기본값은 **L2(유클리드)** 입니다. metric을 명시하지 않으면 정규화 벡터에 대해 의도와 다른 거리(L2)가 적용됩니다. **반드시 `cosine`(또는 `dot`)을 명시**하고, 기본값에 의존하지 마십시오.

| 상황 | 권장 metric | 근거 |
| --- | --- | --- |
| 현재(nomic, norm=1.0) | `cosine` 명시 | cosine=dot 등가, 기본 L2 회피 |
| 모델 교체 후 norm 불보장 | `cosine` + norm 재측정 | 교체 모델(예: BGE-M3)은 정규화 미보장 → **DECISION** |

모델 교체 시 `abs(norm-1.0) < 1e-3` 가드로 정규화 여부를 재측정하고, 깨졌으면 적재 전 1회 정규화하거나 metric 전략을 갱신합니다(→ [01_인덱스_메타데이터_설계.md](./01_인덱스_메타데이터_설계.md)).

## 3. 인덱스 타입 (flat / HNSW)

| 규모 | 인덱스 타입 | 비고 |
| --- | --- | --- |
| 소규모(수천~수만 청크, L1) | flat (정확검색, brute-force) | 재현율 100%, 구현 단순. MVP 기본 |
| 대규모 | HNSW (근사검색) | 속도↑, 재현율 약간↓. 파라미터 튜닝 필요 |

> ChromaDB는 내부적으로 **항상 HNSW 기반**입니다. ChromaDB에서 "flat"이라는 별도 모드는 없으며, 소규모에서는 HNSW가 사실상 정확검색에 가깝게 동작하는 것으로 간주합니다(정확검색 등가 취급).

## 4. 물리 스키마 — 후보별

벡터스토어 선택은 **DECISION**입니다(LanceDB 또는 ChromaDB, 규모 시 Qdrant — [시작프롬프트.md §B-9 #4](../../시작프롬프트.md)). 아래는 후보별 물리 스키마 매핑입니다.

### 4.1 LanceDB (임베디드, 권장 1순위)

PyArrow 스키마로 테이블을 정의합니다. 벡터는 고정 길이 리스트로 선언합니다.

```python
import lancedb
import pyarrow as pa

schema = pa.schema([
    ("chunk_id", pa.string()),
    ("doc_id", pa.string()),
    ("source_path", pa.string()),
    ("chunk_index", pa.int32()),
    ("text", pa.string()),
    ("vector", pa.list_(pa.float32(), 768)),  # 차원 고정(모델 교체 시 1024로 변경)
    ("page", pa.int32()),
    ("section", pa.string()),
])

db = lancedb.connect("data/lancedb")
tbl = db.create_table("chunks", schema=schema, mode="overwrite")

# 적재: vec 은 내부 API embed()의 출력(이미 정규화됨, 재정규화 금지)
def add_chunks(tbl, chunks, vectors):
    rows = []
    for ch, vec in zip(chunks, vectors):
        rows.append({
            "chunk_id": ch.chunk_id,
            "doc_id": ch.doc_id,
            "source_path": ch.source_path,
            "chunk_index": ch.chunk_index,
            "text": ch.text,
            "vector": vec,                      # embed() 결과 그대로
            "page": ch.meta.get("page"),
            "section": ch.meta.get("section"),
        })
    tbl.add(rows)
```

> **인덱스 생성 호출 주의**: LanceDB의 ANN 인덱스 생성 API(`create_index(...)`)는 버전에 따라 시그니처가 다릅니다. 개념적으로는 **HNSW 계열 + cosine** 을 쓰지만, 구체적 호출 형태(예: `index_type`/`metric` 인자명, config 객체 형태)는 **버전 핀 후 [확인필요]** 입니다. 본 문서는 "HNSW(개념)·cosine 명시"까지만 단정하고, 정확한 호출은 설치 버전 문서로 검증하십시오. metric 미지정 시 기본 L2가 적용되는 점(§2)에 유의합니다.

### 4.2 ChromaDB (간편)

컬렉션 생성 시 메타로 거리 공간을 지정합니다. ChromaDB는 항상 HNSW이므로 `hnsw:space`로 metric을 줍니다.

```python
import chromadb

client = chromadb.PersistentClient(path="data/chroma")
collection = client.get_or_create_collection(
    name="chunks",
    metadata={"hnsw:space": "cosine"},   # 기본은 l2 → 정규화 벡터엔 cosine 명시
)

# 적재: ids=chunk_id, embeddings=embed() 결과(재정규화 금지)
def add_chunks(collection, chunks, vectors):
    collection.add(
        ids=[c.chunk_id for c in chunks],
        embeddings=vectors,
        documents=[c.text for c in chunks],
        metadatas=[{
            "doc_id": c.doc_id,
            "source_path": c.source_path,
            "chunk_index": c.chunk_index,
            "page": c.meta.get("page"),
            "section": c.meta.get("section"),
        } for c in chunks],
    )
```

> ChromaDB도 `space` 기본값이 `l2`이므로 `cosine`을 명시해야 합니다(§2 경고와 동일).

## 5. 후보 비교표

| 항목 | LanceDB | ChromaDB |
| --- | --- | --- |
| 형태 | 임베디드(파일 기반) | 임베디드/서버 |
| 인덱스 | flat / HNSW(IVF계열) | 항상 HNSW |
| metric 지정 | 인덱스 생성 시(`metric`, 기본 L2) | 컬렉션 메타 `hnsw:space`(기본 l2) |
| 기본키 | `chunk_id` 컬럼 | `ids`(=chunk_id) |
| 메타 필터 | 컬럼 기반 SQL 유사 | `where` 절 |
| 스키마 | PyArrow 명시 | 동적(메타 dict) |
| 적재 API | `tbl.add(rows)` | `collection.add(...)` |
| 권장 상황 | 명시 스키마·대규모 확장 | 빠른 프로토타이핑 |

> 두 후보 모두 **metric 기본이 L2**라는 공통 함정이 있습니다. 정규화 벡터에는 반드시 cosine을 명시합니다.

## 6. 인덱스 헤더 (모델·차원 메타)

각 인덱스에는 `embedding_model`, `dim`, `normalized`, `metric` 메타(헤더)를 함께 저장해, 질의 임베딩과 인덱스의 모델/차원 불일치를 조기에 차단합니다. **이 헤더의 정의·무효화·재빌드 트리거 메커니즘은 [01_인덱스_메타데이터_설계.md](./01_인덱스_메타데이터_설계.md)가 정본**입니다. 본 문서는 스키마 정의에 한정하고, 헤더 운영은 그 문서를 따릅니다.

| 헤더 키 | 현재값 | 모델 교체(BGE-M3) 시 |
| --- | --- | --- |
| `embedding_model` | `text-embedding-nomic-embed-text-v1.5` | `bge-m3` |
| `dim` | 768 | 1024 (→ 인덱스 전체 재생성) |
| `normalized` | true | **재측정** 필요 |
| `metric` | cosine | cosine(재측정 후 확정) |

→ 모델 교체에 따른 768→1024 재인덱싱 절차는 [../05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md) 참조.

## 7. 요약

- 청크 1건 = 레코드 1건, 기본키 `chunk_id`(내부 API와 합의).
- `vector`는 `embed()` 출력 그대로 저장(재정규화 금지, INV-1).
- metric은 **cosine 명시**(LanceDB/ChromaDB 모두 기본 L2 함정).
- 소규모는 flat, 대규모는 HNSW. ChromaDB는 항상 HNSW.
- 인덱스 헤더 운영은 [01_인덱스_메타데이터_설계.md](./01_인덱스_메타데이터_설계.md)가 정본.
