# 인덱스 메타데이터 설계 (무효화·재빌드 메커니즘)

> 인덱스 헤더에 임베딩 모델 정보를 박아두고, 모델 교체 시 자동으로 인덱스를 무효화·재빌드 트리거하는 **메커니즘**을 설계합니다. 768→1024 차원 교체 리스크를 코드가 아니라 구조로 흡수합니다.
>
> 관련: [시작프롬프트.md](../../시작프롬프트.md) (정본 SSOT) · [03_DATABASE/00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) · [05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md)

## 0. 이 문서의 범위와 SSOT 관계

**역할 분담**(중복·정본 충돌 방지):

- **헤더 필드 스키마의 정본**: [시작프롬프트.md §B-2 ④](../../시작프롬프트.md) 및 [00_벡터스토어_스키마.md §1-1](./00_벡터스토어_스키마.md)이 정의합니다. 필드명·타입·`normalized`/`metric` 의미는 그쪽이 정본이며, 본 문서가 표기를 보여줄 때도 그 정의를 따릅니다.
- **이 문서(`01`)의 책임**: 헤더를 **언제·어떻게 비교해 무효화·재빌드를 트리거하는가**라는 *메커니즘*(상태 머신, fingerprint 비교, 적재·검색 양쪽 게이트)을 정의합니다.
- **실제 재빌드·재인덱싱 실행 절차**: [05_운영_평가/01_운영_런북.md §2](../05_운영_평가/01_운영_런북.md)(임베딩 모델 교체 → 인덱스 전체 재생성)가 담당합니다. 본 문서는 "재빌드가 필요하다"는 *판정*까지만, 그 이후 *실행*은 런북으로 위임합니다.

핵심 사실은 모두 SSOT로 라우팅합니다(리포 공통 규약 `→ §B-N`).

## 1. 왜 인덱스 헤더가 필요한가

docs-em은 1순위 목표가 한국어 검색 품질이고, 현재 임베딩(`text-embedding-nomic-embed-text-v1.5`, 768차원)이 한국어에 약점을 보입니다. "연차 휴가 신청 절차" 질의에서 정답(휴가규정)이 3위로 밀리고 오답(출장경비)보다 점수 격차가 **-0.018**, 즉 Recall@1=0이 실측되었습니다(→ [시작프롬프트.md §A-2 ④](../../시작프롬프트.md)). 이 때문에 교체후보로 BGE-M3(**1024차원**, ctx 8192)가 로드맵 L2에 올라 있습니다(→ [시작프롬프트.md §B-5·§B-12](../../시작프롬프트.md)).

문제의 본질은 **임베딩 모델이 바뀌면 기존 벡터가 전부 무의미해진다**는 점입니다. 768차원 벡터와 1024차원 벡터는 애초에 차원이 달라 같은 인덱스에 공존할 수 없고, 같은 차원이라도 모델이 다르면 벡터 공간이 달라 코사인 유사도가 의미를 잃습니다. 이를 사람이 기억해서 수동으로 재빌드하면 반드시 사고가 납니다.

**해결책**: 인덱스를 만들 때 "어떤 모델로 만든 벡터인지"를 메타데이터 헤더에 박아두고(→ [시작프롬프트.md §B-2 ④](../../시작프롬프트.md)), 검색·적재 시점에 현재 설정과 헤더를 비교해 불일치하면 자동으로 무효화·재빌드를 트리거합니다.

| 헤더 없을 때 | 헤더 있을 때 |
|---|---|
| 모델 교체를 사람이 기억 | 코드가 불일치를 감지 |
| 차원 mismatch가 런타임 에러로 터짐 | 적재 전에 차단·재빌드 |
| 768 벡터 위에 1024 질의 → 조용한 오답 | 헤더 비교로 즉시 거부 |
| "이 인덱스 뭐로 만들었지?" 추적 불가 | 헤더가 출처 기록 |

## 2. 인덱스 헤더 스키마 (참조)

> 헤더 필드의 **정본 정의**는 [00_벡터스토어_스키마.md §1-1](./00_벡터스토어_스키마.md)과 [시작프롬프트.md §B-2 ④](../../시작프롬프트.md)입니다. 아래는 본 문서의 무효화 메커니즘이 참조하는 형태를 그대로 옮긴 것이며, 충돌 시 정본을 따릅니다.

인덱스 1개당 메타데이터 헤더 1개를 둡니다. 핵심 4개 필드(`embedding_model`/`dim`/`normalized`/`metric`)가 모델 정체성을 결정하고, 나머지는 추적·디버깅용입니다.

```json
{
  "schema_version": 1,
  "embedding_model": "text-embedding-nomic-embed-text-v1.5",
  "dim": 768,
  "normalized": true,
  "metric": "cosine",

  "embedding_endpoint": "http://localhost:1234/v1",
  "doc_prefix": "search_document",
  "query_prefix": "search_query",

  "chunker_version": "v1",
  "built_at": "2026-06-16T00:00:00Z",
  "doc_count": 0,
  "vector_count": 0,
  "fingerprint": "<핵심4필드 해시>"
}
```

> 예시값 주석: `doc_count`/`vector_count`는 **빌드 후 채워지는 통계값**(빌드 전 0), `built_at`은 **빌드 시각**(예시는 고정 표기)입니다. `fingerprint`는 §2.2 함수로 빌드 시 계산해 기록합니다.

### 2.1 필드 정의

| 필드 | 값(현재) | 역할 | 무효화 트리거 |
|---|---|---|---|
| `schema_version` | `1` | 헤더 포맷 버전. 헤더 구조 자체가 바뀔 때 증가 | **예** ([추정] 자체 설계, §3.1 코드로 검증) |
| `embedding_model` | `text-embedding-nomic-embed-text-v1.5` | 벡터를 만든 모델 식별자 | **예** (정본) |
| `dim` | `768` | 벡터 차원. nomic=768, BGE-M3=1024 | **예** (정본) |
| `normalized` | `true` | 벡터가 L2 정규화 상태인지 | **예** (정본) |
| `metric` | `cosine` | 거리 함수 | **예**(주의 ↓) |
| `embedding_endpoint` | `http://localhost:1234/v1` | LMStudio OpenAI 호환 엔드포인트 | 아니오(기록용) |
| `doc_prefix` | `search_document` | 적재 시 붙인 prefix | 조건부([추정], §2.2 주의) |
| `query_prefix` | `search_query` | 질의 시 붙인 prefix | 조건부([추정], §2.2 주의) |
| `chunker_version` | `v1` | 청킹 로직 버전 | 경고(재빌드 권장) [추정] 자체 설계 |
| `built_at` | ISO8601 | 인덱스 빌드 시각 | 아니오 |
| `doc_count` / `vector_count` | 정수 | 적재 통계(빌드 후 채움) | 아니오 |
| `fingerprint` | 해시 | 핵심 4필드를 합친 지문 | 비교 키 |

> **`metric` 트리거 주의(과잉 무효화 방지)**: docs-em 실측상 `norm=1.0` 전제에서 `cosine`=`dot`이 등가이므로(→ [시작프롬프트.md §B-7](../../시작프롬프트.md), [00_벡터스토어_스키마.md §2](./00_벡터스토어_스키마.md)), **동일 모델에서 `metric` 라벨만 `cosine`↔`dot`으로 바뀐 경우** 순위는 변하지 않아 전체 재빌드가 불필요합니다. `metric`을 fingerprint에 넣는 것은 **모델 교체 시 정규화 가정이 깨질 수 있는 안전측 설계**이며, `embedding_model`/`dim`이 동일하고 `metric`만 등가 라벨로 달라진 경우는 운영상 재빌드 없이 헤더만 갱신해도 무방합니다(→ 런북 판단).
>
> **`schema_version` 트리거 [추정]**: 헤더 포맷 버전 변경 감지는 SSOT에 없는 자체 설계입니다. §3.1 `check_header()`에서 fingerprint와 **별도로** 비교하도록 구현해야 트리거가 실제로 동작합니다(§3.1 코드 반영).

### 2.2 핵심 4필드 = 인덱스 정체성

`fingerprint`는 무효화 판단을 **단일 비교**로 끝내기 위한 지문입니다. 핵심 4필드만 정규화해 해싱합니다.

```python
import hashlib, json

def index_fingerprint(embedding_model: str, dim: int,
                      normalized: bool, metric: str) -> str:
    """핵심 4필드만으로 인덱스 정체성 지문을 만든다."""
    payload = {
        "embedding_model": embedding_model,
        "dim": dim,
        "normalized": normalized,
        "metric": metric,
    }
    # 키 정렬로 직렬화 안정성 확보
    blob = json.dumps(payload, sort_keys=True, separators=(",", ":"))
    return hashlib.sha256(blob.encode("utf-8")).hexdigest()[:16]
```

`embedding_endpoint`는 fingerprint에서 **제외**합니다. 엔드포인트 포트가 바뀌어도 같은 모델이면 벡터 공간이 동일하기 때문입니다.

**prefix의 fingerprint 포함 여부는 모델 의존입니다 [추정]**:
- 현재 **nomic**은 `search_document`/`search_query` prefix 효과가 미미(코사인 0.687→0.695, 실측 → [시작프롬프트.md §B-2 ③](../../시작프롬프트.md))하므로 fingerprint에서 제외해도 벡터 공간에 영향이 거의 없습니다.
- 그러나 **e5 계열(multilingual-e5 등)은 `query:`/`passage:` prefix가 필수이며 생략 시 품질이 급락**합니다(→ [시작프롬프트.md §B-5](../../시작프롬프트.md)). 이런 모델로 교체하면 prefix가 벡터 공간에 실질 영향을 주므로, 그 경우 prefix를 **fingerprint에 포함하거나 모델 교체와 함께 재빌드**해야 합니다.
- 따라서 "prefix는 항상 fingerprint 제외"는 **nomic 전제의 현재 결정**이며, 모델 교체 시 재검토 대상입니다(아래 부록 [확인필요]). 추적을 위해 헤더에는 항상 기록합니다.

## 3. 무효화·재빌드 트리거 메커니즘 (이 문서의 핵심)

### 3.1 판정 로직

적재(ingest)와 검색(search) **양쪽 진입점**에서 동일한 게이트를 통과시킵니다.

```python
from enum import Enum

class HeaderState(Enum):
    OK = "ok"                  # 일치 — 그대로 사용
    REBUILD = "rebuild"        # 핵심4필드/포맷 불일치 — 무효화·재빌드 필수
    WARN = "warn"              # 청커 등 부가필드 불일치 — 재빌드 권장
    MISSING = "missing"        # 헤더 없음 — 신규 빌드

def check_header(current: dict, stored: dict | None) -> HeaderState:
    # 1) 헤더 자체가 없음(신규 인덱스)
    if stored is None:
        return HeaderState.MISSING

    # 2) 헤더 포맷 버전 불일치 → 헤더 해석을 신뢰할 수 없으므로 재빌드
    #    [추정] 자체 설계. §2.1 schema_version 트리거를 코드로 실제 검증.
    if current.get("schema_version") != stored.get("schema_version"):
        return HeaderState.REBUILD

    # 3) 핵심 4필드 지문 비교
    cur_fp = index_fingerprint(
        current["embedding_model"], current["dim"],
        current["normalized"], current["metric"])
    stored_fp = stored.get("fingerprint")
    if stored_fp is None:
        # 레거시 헤더(fingerprint 키 없음): MISSING과 구분되는 별도 케이스.
        # 정체성을 확인할 수 없으므로 안전측으로 REBUILD 처리한다.
        return HeaderState.REBUILD
    if cur_fp != stored_fp:
        return HeaderState.REBUILD          # 모델/차원/정규화/메트릭 변경

    # 4) 부가필드(청킹) 불일치 — 권장 재빌드 [추정] 자체 설계
    if current.get("chunker_version") != stored.get("chunker_version"):
        return HeaderState.WARN

    return HeaderState.OK
```

> 케이스 구분 명시:
> - **`MISSING`** = 헤더 파일/메타 자체가 없음(신규 인덱스).
> - **레거시 헤더(`fingerprint` 키 없음)** = 헤더는 있으나 정체성 지문을 알 수 없음 → 안전측 `REBUILD`. `MISSING`과 의도적으로 구분합니다.
> - **`schema_version` 불일치** = 헤더 포맷이 달라 필드 해석을 신뢰할 수 없음 → `REBUILD`(코드 2단계로 실제 검증).

### 3.2 상태별 동작

| 상태 | 의미 | 적재 시 동작 | 검색 시 동작 |
|---|---|---|---|
| `OK` | 헤더 일치 | 증분 적재 진행 | 정상 검색 |
| `MISSING` | 헤더 없음(신규) | 새 헤더로 빌드 | **차단** → 빌드 안내 |
| `REBUILD` | 핵심4필드/포맷 불일치·레거시 | 기존 인덱스 폐기 후 전체 재빌드(→ 런북 §2) | **차단** → 재빌드 요구 |
| `WARN` | 청커 등 변경 | 경고 로그, 재빌드 권장 | 경고 후 검색 허용 |

검색 시 `REBUILD`/`MISSING`을 **반드시 차단**하는 이유: 768차원 인덱스에 1024차원 질의 벡터를 던지면 벡터스토어가 런타임 에러를 내거나, 운 나쁘면 차원이 우연히 맞아 **조용한 오답**(silent wrong)을 반환합니다. 헤더 게이트가 이 경로를 적재·검색 양쪽에서 원천 차단합니다(→ [00_벡터스토어_스키마.md §8](./00_벡터스토어_스키마.md), [시작프롬프트.md §B-2 ④](../../시작프롬프트.md)).

### 3.3 768 → 1024 교체 시나리오

BGE-M3로 교체할 때 일어나는 **판정** 흐름입니다. 실제 재임베딩·재인덱싱 **실행 절차는 런북이 담당**합니다(→ [05_운영_평가/01_운영_런북.md §2 임베딩 모델 교체 → 인덱스 전체 재생성](../05_운영_평가/01_운영_런북.md)). 다운로드·교체 시점은 **DECISION** — 사용자 결정(GGUF 1~2GB).

```
[현재 설정] embedding_model=BGE-M3, dim=1024,
            normalized=[확인필요], metric=[확인필요](cosine 후보, norm 재측정 후 확정)
                    │
                    ▼ index_fingerprint() 재계산
        cur_fp = "a1b2..."  ≠  stored.fingerprint = "9f3e..."(nomic/768)
                    │
                    ▼ check_header()
              HeaderState.REBUILD
                    │
        ┌───────────┴───────────┐
        ▼ (적재 경로)            ▼ (검색 경로)
  기존 인덱스 폐기          검색 차단 + 메시지:
  → 런북 §2 절차로          "임베딩 모델이 교체되어
     전체 재임베딩(1024)      인덱스 재빌드가 필요합니다"
  → 새 헤더 기록(dim=1024,
     normalized=재측정값)
```

핵심: **사람이 768/1024를 의식할 필요가 없습니다.** 설정에서 모델만 바꾸면 fingerprint가 자동으로 달라지고, 게이트가 재빌드를 강제합니다. 실제 폐기·재임베딩·재인덱싱·롤백은 런북 §2의 절차로 수행합니다.

> **차원만이 아니라 컨텍스트도 바뀝니다**: nomic과 BGE-M3는 차원(768→1024) 외에 컨텍스트 길이도 다릅니다. BGE-M3 ctx는 **8192**(실측 → [시작프롬프트.md §B-12·§B-5](../../시작프롬프트.md))로, 청크 길이·입력 가드 한계가 달라질 수 있습니다. 교체 시 청킹 파라미터 영향은 런북 §2와 §B-2에서 함께 검토합니다.
>
> **`normalized`/`metric`은 [확인필요]**: nomic은 출력이 이미 L2 정규화(norm=1.0)되어 재정규화 금지지만, BGE-M3는 서빙 방식에 따라 norm을 보장하지 않을 수 있습니다(→ [시작프롬프트.md §B-7](../../시작프롬프트.md), [00_벡터스토어_스키마.md §1-1·§7](./00_벡터스토어_스키마.md)). 교체 시 **임베딩 1건을 뽑아 L2 norm을 재측정**해 `normalized` 헤더를 확정하고, 그 결과에 따라 `metric` 등가성(`cosine`=`dot`)이 성립하는지 재판단해야 합니다. **재측정 전까지 `normalized`를 신뢰하면 안 됩니다(A-2 ①).** 이 값이 틀리면 코사인 유사도가 왜곡됩니다.

## 4. 헤더 저장 위치

벡터스토어 선택(LanceDB vs ChromaDB)은 아직 미확정입니다(→ [03_DATABASE/00_벡터스토어_스키마.md](./00_벡터스토어_스키마.md) 참조, **DECISION**). 두 경우 모두 헤더를 둘 자리가 있습니다.

| 스토어 | 헤더 저장 방식 | 비고 |
|---|---|---|
| LanceDB | 별도 메타 테이블 또는 사이드카 `_index_header.json` | 테이블 자체가 헤더 슬롯을 제공하지 않음. 테이블 메타 API는 [확인필요](→ 스키마.md §1-2) |
| ChromaDB | 컬렉션 `metadata`(dict)에 **스칼라 핵심필드만** 동봉 | metadata는 `str/int/float/bool`만 허용(중첩 불가) — §5 주의 |
| 공통 폴백 | 인덱스 디렉터리 사이드카 `_index_header.json` | 스토어 무관, **권장** |

**권장**: 스토어에 종속되지 않도록 사이드카 `_index_header.json`을 1차 채택합니다(**DECISION** — 스토어 확정 후 재검토). 사이드카 파일명은 형제 문서와 **단일화**하여 `_index_header.json`으로 고정합니다(→ [00_벡터스토어_스키마.md §1-2 L120](./00_벡터스토어_스키마.md), 예: `data/lancedb/_index_header.json`). 스토어 내장 메타와 사이드카가 둘 다 있으면 **사이드카를 정본**으로 봅니다.

```
data/lancedb/  (또는 data/chroma/)
├── _index_header.json       # ← 헤더(정본, 형제 문서와 동일 파일명)
├── lance.table/  또는  chroma/
└── bm25/                    # BM25 인덱스(별도 무효화 규칙 — §5)
```

## 5. 하이브리드 검색에서의 확장

docs-em은 L2에서 BM25(Kiwi 형태소 기반) + 벡터를 RRF로 결합하는 하이브리드를 계획합니다(→ [시작프롬프트.md §B-6](../../시작프롬프트.md)). BM25 인덱스는 **임베딩 모델과 무관**하므로 무효화 규칙이 다릅니다.

| 서브인덱스 | 무효화 트리거 | 비고 |
|---|---|---|
| 벡터 인덱스 | 핵심4필드(embedding_model/dim/normalized/metric) | 본 문서 메인 |
| BM25 인덱스 | `tokenizer_version`(Kiwi 버전) + `chunker_version` | 임베딩 교체와 무관 [추정] 자체 설계 |

**[추정] 자체 설계**: nomic→BGE-M3 교체는 임베딩 벡터 공간만 바꾸므로 **벡터 인덱스만 재빌드하면 되고 BM25는 그대로 둘 수 있다**는 것은 SSOT에 명문화되지 않은 설계 주장입니다(다만 임베딩 교체가 BM25 토큰화·청킹과 독립이라는 점에서 합리적). `chunker_version`이 함께 바뀌면 BM25도 재빌드 대상입니다. 헤더에 서브인덱스별 트리거를 분리해 두면 불필요한 전체 재빌드를 피합니다.

```json
{
  "vector": { "fingerprint": "9f3e...", "dim": 768 },
  "bm25":   { "tokenizer_version": "kiwi-x.y", "chunker_version": "v1" }
}
```

> **저장 위치 주의(타입 제약)**: 위와 같은 **중첩(nested) dict 헤더는 ChromaDB `metadata`에 저장할 수 없습니다**. ChromaDB metadata는 스칼라(`str/int/float/bool`)만 허용하기 때문입니다(→ [00_벡터스토어_스키마.md §163·§179](./00_벡터스토어_스키마.md)). 따라서 **중첩 구조 헤더는 사이드카 `_index_header.json`(json) 전용**이며, ChromaDB 컬렉션 metadata에는 핵심필드를 **평탄화한 스칼라**(예: `vector_fingerprint`, `vector_dim`, `bm25_tokenizer_version`)로만 동봉합니다. 정본은 어느 경우든 사이드카입니다(§4).

> 리랭커(BGE-reranker-v2-m3)는 인덱스를 만들지 않고 질의 시점에만 동작하므로 헤더 무효화 대상이 아닙니다(→ [시작프롬프트.md §B-2 ⑥](../../시작프롬프트.md), 리랭킹 문서가 담당).

## 6. 설계 원칙 요약

- **헤더 필드 정본은 SSOT/스키마 문서**: 필드 정의는 [시작프롬프트.md §B-2 ④](../../시작프롬프트.md)·[00_벡터스토어_스키마.md §1-1](./00_벡터스토어_스키마.md), 본 문서는 무효화 *메커니즘* 담당, 재빌드 *실행*은 [런북 §2](../05_운영_평가/01_운영_런북.md).
- **정체성은 4필드로 충분**: `embedding_model`/`dim`/`normalized`/`metric`이 벡터 공간을 결정합니다. 나머지는 추적용.
- **fingerprint 단일 비교**: 4필드를 해시 1개로 압축해 게이트 판정을 단순화합니다(단, `schema_version`은 별도 비교).
- **양쪽 진입점 게이트**: 적재와 검색 모두에서 헤더를 검사해 조용한 오답을 원천 차단합니다.
- **검색은 불일치 시 차단**: 차원 mismatch로 인한 런타임 에러·silent wrong을 빌드 전에 막습니다.
- **사이드카 정본**: 스토어 종속을 피해 `_index_header.json`을 SSOT로 둡니다(형제 문서와 파일명 단일화, **DECISION** — 스토어 확정 후 재검토). 중첩 헤더는 사이드카 전용(ChromaDB metadata는 스칼라만).
- **서브인덱스 분리 [추정]**: 벡터/BM25 무효화 트리거를 분리해 불필요한 재빌드를 피합니다.

이 설계로 768→1024 교체는 "설정에서 모델명만 변경"으로 끝나고(실행은 런북 §2), 차원·컨텍스트 변경의 위험은 사람의 기억이 아니라 구조가 흡수합니다.
