## 2. 마일스톤 로드맵 (L1 → L2 → L3)

큰 그림은 → [시작프롬프트.md](../../시작프롬프트.md) §B-10 참조. 아래는 각 레벨을 산출물·완료기준으로 구체화한 것입니다.

```
L1 검색 MVP + 평가 하네스
        │  (게이트 G1: 회귀케이스 측정 가능 + 베이스라인 수치 확정)
        ▼
L2 임베딩 교체 + 하이브리드 + reasoning 폴백 게이트
        │  (게이트 G2: 회귀케이스 Recall@1 회복 + content 빈값 0건)
        ▼
L3 리랭킹 · 로컬 서빙 · 자동화
           (게이트 G3: 리랭킹 반영 지표 개선 + 재현 자동화)
```

## 3. L1 — 검색 MVP + 평가 하네스

목표: "현행 nomic 임베딩으로 한국어 검색이 실제로 얼마나 약한지"를 수치로 고정하고, 이후 모든 개선을 비교할 베이스라인을 만든다.

### 3.1 산출물

| 산출물 | 설명 |
|--------|------|
| 적재 파이프라인 | PDF/DOCX → 텍스트 추출 → 문장분리(kss) → 청킹 |
| 임베딩 연동 | text-embedding-nomic-embed-text-v1.5(768차원, **출력이 이미 L2 정규화됨 norm=1.0 → 재정규화 금지**) |
| 벡터 인덱스 | LanceDB 또는 ChromaDB **DECISION** (스키마 정본: [03_DATABASE/00_벡터스토어_스키마.md](../03_DATABASE/00_벡터스토어_스키마.md)) |
| 평가 하네스 | 골든셋 로딩 → 검색 → Recall@k/MRR/nDCG 산출 |
| 베이스라인 리포트 | 회귀케이스 포함 현행 수치 스냅샷 |

### 3.2 완료기준 (게이트 G1)

- [ ] 골든셋으로 Recall@1/3/5/10, MRR, nDCG 자동 산출 가능
- [ ] 회귀케이스("연차 휴가 신청 절차") 재현: 정답(휴가규정) **3위**, 오답(출장경비) 대비 격차 **-0.018**, **Recall@1=0** 확인
- [ ] prefix(search_document/query) 효과 검증: **0.687 → 0.695** 미미함 기록
- [ ] 베이스라인 수치가 문서로 고정되어 L2 비교 기준 성립

> L1은 "고치는" 단계가 아니라 "정확히 측정하는" 단계입니다. 회귀케이스의 실패를 재현해야 게이트를 통과합니다.

### 3.3 주의

- 임베딩 출력은 norm=1.0으로 이미 정규화되어 있습니다. 코사인 유사도 계산 전 재정규화하면 안 됩니다(SSOT §B-2 ③·§B-7).
- `usage.prompt_tokens=0` 오보고가 있으므로 토큰 기반 청크 산정에 이 값을 신뢰하지 마십시오. 토큰 카운트는 자체 계산으로 대체합니다. → 운영 절차는 [05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md) 참조

## 4. L2 — 임베딩 교체 + 하이브리드 + reasoning 폴백 게이트

목표: 한국어 검색 약점을 실제로 해소한다. 임베딩 교체와 하이브리드로 회귀케이스 Recall@1을 회복하고, 생성 단계의 BLOCKING(빈 content) 이슈를 게이트로 차단한다.

### 4.1 산출물

| 산출물 | 설명 |
|--------|------|
| 임베딩 교체 실험 | nomic ↔ BGE-M3(1024차원, ctx 8192[실측], Dense+Sparse+ColBERT, MIT) 동일 골든셋 비교. 가중치 다운로드는 **DECISION** |
| **인덱스 재생성 세트** | 차원 768→1024 전환 시 **인덱스 전체 재생성 + norm 재측정 + INV-1(norm 가드) 갱신**(SSOT §B-5, [99_AI참조/01_실측제약_불변식.md](../99_AI참조/01_실측제약_불변식.md) §8). 이것이 임베딩 교체의 최대 비용 |
| BM25 경로 | Kiwi 형태소 분석 기반 토큰화 + rank_bm25 |
| 하이브리드 결합 | 벡터 + BM25 → RRF로 순위 융합 |
| reasoning 폴백 게이트 | `message.content` 빈값 시 `reasoning_content` 폴백 + `<think>` 정제 + `length` 잘림 재시도 |
| L2 비교 리포트 | L1 베이스라인 대비 변수별(임베딩/BM25/RRF) 단일 변경 측정 |

### 4.2 reasoning 폴백 게이트 — 처리 규칙

세 생성모델(qwen/qwen3.6-35b-a3b[35B MoE, ctx 8192 실측], qwen/qwen3.5-9b[9B는 추정, ctx 8192 실측], google/gemma-4-e4b[4B는 추정, ctx 65536 실측]) 공통으로 `message.content`가 빈 문자열로 오고 답이 `message.reasoning_content`에 담깁니다. `/no_think`·`enable_thinking=false`·`reasoning_effort=low`·`max_tokens=4000`은 **전부 무효**이며, 무한사고로 `finish_reason=length` 잘림이 발생합니다(한 문장 질문에 reasoning **12,972자** 실측).

> 아래는 게이트의 **개념 예시**입니다. **구현 정본은 [시작프롬프트.md](../../시작프롬프트.md) §B-7**(`_clean_reasoning` 정제 함수 + `chat()` 래퍼의 인라인 재귀 재시도)이며, 실제 코드는 정본을 그대로 따릅니다. 정본은 `length`일 때 **먼저 재귀 재시도하고 그 반환을 정제·사용**하는 순서입니다(정제 결과를 폐기하지 않음).

```python
# 개념 예시 — 정본 구현은 시작프롬프트.md §B-7 (_clean_reasoning + chat() 재귀 재시도)
def extract_answer(resp, *, max_tokens, _retry=True):
    msg = resp.choices[0].message
    # content 빈값 → reasoning_content 폴백 (SSOT §B-4·§B-12)
    raw = msg.content or getattr(msg, "reasoning_content", "") or ""
    # length 잘림이면 먼저 재귀 재시도 → 그 결과를 사용 (정본 순서와 동일)
    if resp.choices[0].finish_reason == "length" and _retry:
        resp = chat(..., max_tokens=max_tokens * 3, _retry=False)  # 인라인 재귀
        return extract_answer(resp, max_tokens=max_tokens * 3, _retry=False)
    return _clean_reasoning(raw)   # <think>...</think> 정제 (정본: 시작프롬프트.md §B-7)
```

### 4.3 완료기준 (게이트 G2)

- [ ] 회귀케이스 Recall@1 회복: "연차 휴가 신청 절차" → 정답(휴가규정) **1위** 달성
- [ ] 임베딩/BM25/RRF 각 변수의 기여도가 단일 변경 측정으로 기록됨
- [ ] 임베딩 교체 채택 시 **인덱스 전체 재생성 완료 + norm 재측정값으로 INV-1 가드 갱신** 확인
- [ ] 생성 응답에서 빈 content로 인한 답변 누락 **0건** (폴백 게이트 적용)
- [ ] `finish_reason=length` 잘림 케이스에서 재시도로 완결 답변 확보

### 4.4 주의

- 변수는 **한 번에 하나만** 바꿔 측정합니다(임베딩 ↔ BM25 ↔ RRF 동시 변경 금지). → 절차 정본: [시작프롬프트.md](../../시작프롬프트.md) §B-8-3
- BGE-M3는 Dense/Sparse/ColBERT 다중 표현이 가능하나, L2에서는 Dense 중심으로 시작하고 Sparse/ColBERT 활용은 L3 또는 후속으로 미룹니다 [추정].
- **prefix 규칙은 모델별로 다릅니다.** nomic은 prefix 효과가 미미했으나(0.687→0.695), **e5 계열 등 다른 모델로 교체하면 `query:`/`passage:` prefix가 실제로 효과가 있어 반드시 함께 교체해야 합니다. nomic 경험을 그대로 이식해 prefix를 생략하면 품질이 급락합니다.** BGE-M3의 prefix 요구사항은 모델 카드로 확인 `[확인필요]`. → [99_AI참조/01_실측제약_불변식.md](../99_AI참조/01_실측제약_불변식.md) §6, [05_운영_평가/01_운영_런북.md](../05_운영_평가/01_운영_런북.md) §2.1, SSOT §B-5

## 5. L3 — 리랭킹 · 로컬 서빙 · 자동화

목표: 검색 상위 결과의 정밀도를 리랭커로 끌어올리고, 파이프라인을 로컬 서빙·자동 평가로 운영 가능한 형태로 마감한다.

### 5.1 산출물

| 산출물 | 설명 |
|--------|------|
| 리랭킹 단계 | BGE-reranker-v2-m3 적용(하이브리드 후보 → 재정렬). 로컬 서빙 방식(LMStudio 호환 여부) **[확인필요]** |
| 로컬 서빙 | 검색·생성 파이프라인을 호출 가능한 로컬 서비스로 노출 |
| 자동화 | 골든셋 평가 자동 실행 + 리포트 생성(회귀 가드) |
| L3 종합 리포트 | L1→L2→L3 지표 추이 + 회귀케이스 최종 상태 |

### 5.2 완료기준 (게이트 G3)

- [ ] 리랭킹 적용 후 Recall@k/MRR/nDCG가 L2 대비 개선됨 — **정량 임계: nDCG@10이 L2 대비 개선(저하 없음), 회귀케이스 저하 허용치 0** (최소 개선폭·허용 회귀 범위의 정본은 SSOT §B-8 평가 기준에 연결)
- [ ] 회귀케이스가 리랭킹 포함 파이프라인에서도 **Recall@1 유지**
- [ ] 골든셋 평가가 **1커맨드로 재현**되고 리포트가 자동 생성됨
- [ ] BGE-reranker-v2-m3 **로컬 서빙 방식이 확정되어 문서화됨** (`[확인필요]` 해소)

## 6. 의존성

### 6.1 마일스톤 간 의존성

```
L1(평가 하네스 + 베이스라인) ──선행─▶ L2(개선 측정의 기준)
L2(하이브리드 후보군)        ──선행─▶ L3(리랭커 입력)
L1(평가 하네스)              ──재사용─▶ L2, L3 (동일 골든셋·지표)
```

- L2의 "개선 여부"는 L1 베이스라인 없이는 판정 불가 → L1이 L2의 절대 선행.
- L3 리랭커는 하이브리드 후보군(L2)을 입력으로 받음 → L2가 L3의 절대 선행.

### 6.2 외부 의존성

| 의존 대상 | 상태 | 영향 |
|-----------|------|------|
| LMStudio 로컬 서버 가동(`:1234`) | 필수 | 전 단계 차단 시 진행 불가 |
| 임베딩 모델 text-embedding-nomic-embed-text-v1.5 | 가동 중 | L1 베이스라인 |
| BGE-M3 GGUF(1~2GB) 다운로드 | **DECISION** | L2 임베딩 교체 가부 |
| 생성모델 3종 로컬 가동(`parallel=4`) | 가동 중 | reasoning 폴백 검증 |
| 기본 생성모델 선택 | **DECISION** | gemma-4-e4b 권장 — reasoning 폴백 게이트의 기준 모델 |
