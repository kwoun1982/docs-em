# 01. 유스케이스 · 사용자 스토리

> docs-em RAG 시스템의 주요 시나리오, 액터, 사용자 스토리(As-a/I-want/So-that), 수용기준을 정의합니다. "연차 휴가" 회귀 케이스를 1급 시민으로 다룹니다.
> 관련: [상위문서] [시작프롬프트.md](../../시작프롬프트.md) §A-2 · §B-12 (정본 SSOT)

## 0. 이 문서를 읽는 법

- 본 문서의 모든 실측 수치(768차원, 1024차원, -0.018, 12,972자, 0.687→0.695, prompt_tokens=0 등)와 모델 사양은 SSOT(`시작프롬프트.md`) §A-2 · §B-12에서 도출했습니다. 두 곳과 한 글자라도 모순되면 **SSOT가 정본**이며 이 문서를 고칩니다.
- 태그 의미:
  - `[추정]` = 미검증 가정 (예: qwen3.5-9b/gemma-4-e4b의 파라미터 수)
  - `[확인필요]` = 미확인 사실 (예: 스택 최종 확정, BGE-M3 norm 보장 여부)
  - **DECISION** = 사용자가 결정해야 하는 항목 (예: BGE-M3 다운로드·교체)
- 본 문서는 "무엇을 누가 왜"(요구사항)만 정의합니다. 구현 코드·상세 설계·평가 임계치 확정은 본 문서가 맡지 않으며 실재하는 링크로만 위임합니다.
- 실측 시점: **실측일 2026-06-16** (§6 베이스라인 표 기준).

## 1. 개요

본 문서는 docs-em의 **무엇을 누가 왜** 사용하는지를 유스케이스와 사용자 스토리 형태로 명세합니다. 시스템의 핵심 목적은 **한국어 사내문서 검색 품질**이며(→ [시작프롬프트.md](../../시작프롬프트.md) §A-2 #4), 모든 스토리는 이 목적에 종속됩니다. 아키텍처·구현 방식은 본 문서의 범위가 아니며 링크로만 참조합니다(→ [04_아키텍처_API/](../04_아키텍처_API/)).

전제(불변식 요약):

- 모든 추론·임베딩은 **로컬 LMStudio**(`http://localhost:1234/v1`, OpenAI 호환, api_key 더미)에서 수행되며, 외부 클라우드 API(OpenAI/Anthropic/Cohere)는 사용하지 않습니다. → [시작프롬프트.md](../../시작프롬프트.md) §A-1 · §B-12
- 현재는 코드가 없는 **문서화 단계**이므로, 본 스토리들은 구현 대상의 요구사항 정의서 역할을 합니다.

> 상위 PRD(액터 권한·페르소나·전사 목표) 문서는 현재 미작성입니다. 본 문서가 필요로 하는 액터 권한·목표 임계치는 §2·§3·§7에서 실재하는 위치로 재지정했습니다. PRD 신설 여부는 [확인필요].

## 2. 액터(Actors)

| 액터 | 유형 | 설명 | 주요 관심사 |
| --- | --- | --- | --- |
| **일반 사용자(질의자)** | 1차/인간 | 사내문서에 대해 자연어로 질문하는 직원 | 정확한 답·근거 출처·빠른 응답 |
| **운영자/관리자** | 2차/인간 | 문서 인덱싱, 골든셋 관리, 평가 지표 모니터링 담당 | 검색 품질 지표, 회귀 방지 |
| **평가자(QA)** | 2차/인간 | 골든셋 기반 Recall@k·MRR·nDCG 측정·회귀 검출 | 측정 재현성, 변수 1개씩 변경 |
| **LMStudio 추론 엔진** | 2차/시스템 | 임베딩·생성모델 서빙(`localhost:1234`) | 가용성, 응답 형식 일관성 |
| **인덱싱 파이프라인** | 2차/시스템 | 문서 파싱·청킹·임베딩·저장 자동화 | 파싱 정확도, 멱등성 |

> 액터별 상세 권한·페르소나는 본 문서 범위 밖입니다. 별도 PRD가 신설되기 전까지는 정의하지 않으며, 시스템 동작 제약은 [시작프롬프트.md](../../시작프롬프트.md) §A-1을 정본으로 따릅니다. [확인필요]

## 3. 주요 유스케이스(Use Cases)

```
                ┌─────────────────────────────────────────┐
                │            docs-em RAG 시스템            │
                │                                         │
  [일반 사용자]──┼──▶ UC-1 사내문서 질의응답               │
                │      └─ UC-1a 근거 출처 확인            │
                │                                         │
  [운영자]──────┼──▶ UC-2 문서 인덱싱/재색인              │
                │                                         │
  [평가자]──────┼──▶ UC-3 검색 품질 평가(골든셋)          │
                │      └─ UC-3a 연차휴가 회귀 검증        │
                │                                         │
                │   (모든 추론 ◀─▶ [LMStudio])           │
                └─────────────────────────────────────────┘
```

> 액터 권한 모델(누가 인덱싱/평가를 실행할 수 있는가)은 본 문서에서 정의하지 않습니다. 권한 분리가 필요해지면 별도 PRD에서 다룹니다. [확인필요]

### UC-1. 사내문서 질의응답 (핵심)

- **주 액터**: 일반 사용자
- **목표**: 자연어 질문에 대해, 사내문서를 근거로 한 답변과 출처를 받는다.
- **사전조건**: 대상 문서가 인덱싱되어 있고 LMStudio가 가용하다.
- **기본 흐름**:
  1. 사용자가 자연어 질의를 입력한다 (예: "연차 휴가 신청 절차").
  2. 시스템이 질의를 임베딩(`text-embedding-nomic-embed-text-v1.5`, 768차원, 출력 이미 L2 정규화됨 → 재정규화 금지)하고 검색한다. → [시작프롬프트.md](../../시작프롬프트.md) §A-2 #1 · §B-12
  3. 상위 k개 청크를 근거로 생성모델이 답변을 작성한다.
  4. 답변과 함께 출처 문서·청크를 제시한다.
- **대안/예외 흐름**:
  - 생성모델의 `message.content`가 빈 문자열이면 `message.reasoning_content`로 폴백하고 `<think>` 블록을 정제한다. → §5.2, [시작프롬프트.md](../../시작프롬프트.md) §B-4(선결 Blocking) · §B-7(래퍼 코드 정본)
  - `finish_reason=length`(무한사고 잘림) 시 더 높은 `max_tokens`로 1회 재시도한다. → [시작프롬프트.md](../../시작프롬프트.md) §B-4 · §B-7
  - 검색 결과 신뢰도가 낮으면(아래 AC-1.4의 판정 기준 적용) "확실하지 않습니다" 취지로 응답하고 근거 부족을 명시한다.

### UC-1a. 근거 출처 확인

- 사용자가 답변에 인용된 출처(문서명·청크)를 열어 원문을 확인한다. 출처 없는 답변은 신뢰 불가로 간주한다.

### UC-2. 문서 인덱싱 / 재색인

- **주 액터**: 운영자
- **목표**: 신규/변경 문서를 파싱·청킹·임베딩하여 벡터스토어에 반영한다.
- **흐름**: 문서 수집(pymupdf/python-docx) → 문장분리(kss)·형태소(Kiwi) → 청킹 → 임베딩 → 저장. → [04_아키텍처_API/](../04_아키텍처_API/) · [시작프롬프트.md](../../시작프롬프트.md) §B-2(데이터 파이프라인) 참조(구현 상세는 본 문서 범위 밖).
- **수용기준 요약**: 동일 입력 재색인 시 멱등(중복 청크 미생성, 동일성 키 정의는 AC-5.2), 파싱 실패 문서는 스킵·로그.

### UC-3. 검색 품질 평가 (골든셋)

- **주 액터**: 평가자
- **목표**: 골든셋으로 Recall@k(1/3/5/10)·MRR·nDCG를 측정하고 회귀를 검출한다. → [시작프롬프트.md](../../시작프롬프트.md) §B-8(평가 방법 정본) · [05_운영_평가/](../05_운영_평가/)
- **원칙**: 변수는 **1개씩만** 변경하여 측정한다(→ [시작프롬프트.md](../../시작프롬프트.md) §B-8-3).

### UC-3a. 연차휴가 회귀 검증 (필수 회귀 케이스)

- 골든셋에 "연차 휴가 신청 절차" 질의가 회귀 케이스로 포함된다(→ [시작프롬프트.md](../../시작프롬프트.md) §B-8-1). 현재 베이스라인은 **Recall@1=0**(아래 §6 실측). 임베딩 교체(nomic 768차원 → BGE-M3 1024차원)·하이브리드 적용 후 이 케이스의 **Recall@1=1**(정답 1위) 달성 여부를 게이트로 삼는다. → [시작프롬프트.md](../../시작프롬프트.md) §B-5 · §B-8-3

## 4. 사용자 스토리(User Stories)

각 스토리는 **As-a / I-want / So-that** 형식이며, 우선순위(P0=필수)와 수용기준(AC)을 동반합니다. 로드맵 단계(L1/L2/L3)는 → [시작프롬프트.md](../../시작프롬프트.md) §B-10.

### US-1 (P0) — 자연어 질의응답 · L1

- **As a** 일반 사용자,
- **I want** "연차 휴가 신청 절차" 같은 한국어 질문을 입력하면,
- **So that** 관련 사내문서(예: 휴가규정)를 근거로 한 답변과 출처를 받고 싶다.

**Acceptance Criteria**

| # | 조건 | 기대 결과 |
| --- | --- | --- |
| AC-1.1 | 질의 입력 시 | 상위 k개 근거 청크와 출처 문서명이 함께 제시된다 |
| AC-1.2 | 답변 생성 시 | 생성모델 응답의 `content`가 비어 있으면 `reasoning_content` 폴백으로 비어 있지 않은 답이 반환된다 (정본 → [시작프롬프트.md](../../시작프롬프트.md) §B-4 · §B-7) |
| AC-1.3 | `<think>` 포함 응답 | `<think>…</think>` 블록이 제거된 정제 답변만 사용자에게 노출된다 (정본 → §B-4 ② · §B-7) |
| AC-1.4 | 근거 부족 시 | "확실하지 않습니다" 취지로 응답하고 추측 답변을 만들지 않는다. **판정 기준**: 상위 1위 청크가 없거나, 검색 점수가 컷오프 미만인 경우를 "근거 부족"으로 본다. 컷오프 점수의 구체값은 골든셋 기반으로 → [05_운영_평가/](../05_운영_평가/) · [시작프롬프트.md](../../시작프롬프트.md) §B-8에서 확정한다. [확인필요] |

### US-2 (P0) — 한국어 검색 정확도 (연차휴가 회귀) · L1/L2

- **As a** 일반 사용자,
- **I want** "연차 휴가 신청 절차"를 물었을 때,
- **So that** 출장경비 같은 **오답 문서가 아니라** 휴가규정이 1순위로 검색되길 원한다.

**Acceptance Criteria**

| # | 조건 | 기대 결과 |
| --- | --- | --- |
| AC-2.1 | 베이스라인(nomic) 측정 | 정답(휴가규정) 순위·오답(출장경비)과의 점수 격차가 골든셋에 기록된다 (현재 실측: 3위, 격차 -0.018 → §6) |
| AC-2.2 | 개선 게이트 | 임베딩(768→1024차원, BGE-M3)/하이브리드 변경 후 이 케이스의 Recall@1이 0→1로 개선되거나, 미개선 시 사유가 기록된다 |
| AC-2.3 | 측정 절차 | 변수 1개씩만 변경하여 before/after를 비교한다 (→ §B-8-3) |

> 베이스라인 실측 수치는 §6 참조. 목표 임계치 확정은 → [시작프롬프트.md](../../시작프롬프트.md) §B-8 · [05_운영_평가/](../05_운영_평가/)에서 다룹니다.

### US-3 (P0) — 출처 표기 · L1

- **As a** 일반 사용자,
- **I want** 답변에 인용된 원문 출처를 확인하고,
- **So that** 답변의 신뢰성을 검증하고 원문으로 이동할 수 있길 원한다.

**Acceptance Criteria**

| # | 조건 | 기대 결과 |
| --- | --- | --- |
| AC-3.1 | 답변 표시 | 각 답변에 1개 이상의 출처(문서명·청크 식별자)가 붙는다 |
| AC-3.2 | 출처 부재 | 근거 청크가 없으면(상위 1위 청크 부재 또는 점수 컷오프 미만, AC-1.4와 동일 기준) 답변 대신 근거 부족을 알린다 |

### US-4 (P1) — 응답 안정성(무한사고 방어) · L2

- **As a** 일반 사용자,
- **I want** 한 문장짜리 질문에도 답이 잘려서 오지 않길 바라고,
- **So that** 끝까지 완결된 답변을 받고 싶다.

**Acceptance Criteria**

| # | 조건 | 기대 결과 |
| --- | --- | --- |
| AC-4.1 | `finish_reason=length` | 더 높은 `max_tokens`로 재시도하는 로직이 동작하여 잘리지 않은 답을 반환한다 (정본 → [시작프롬프트.md](../../시작프롬프트.md) §B-4 ③ · §B-7) |
| AC-4.2 | 폴백 우선순위 | `content` → `reasoning_content` 폴백 → `<think>` 정제 → `length` 재시도 순서가 보장된다 (정본 → §B-4 · §B-7) |

> 무한사고 잘림의 원인·실측(한 문장 질문에 reasoning 12,972자, `/no_think`·`enable_thinking=false`·`reasoning_effort=low`·`max_tokens=4000` 전부 무효)은 → [시작프롬프트.md](../../시작프롬프트.md) §B-4.

### US-5 (P0) — 문서 인덱싱 · L1

- **As a** 운영자,
- **I want** 신규 사내문서(.pdf/.docx)를 색인에 추가하고,
- **So that** 사용자가 그 내용을 검색할 수 있길 원한다.

**Acceptance Criteria**

| # | 조건 | 기대 결과 |
| --- | --- | --- |
| AC-5.1 | 색인 실행 | 파싱→청킹→임베딩→저장이 완료되고 처리 문서 수가 보고된다 |
| AC-5.2 | 재색인(멱등) | 동일 입력 재색인 시 중복 청크가 생성되지 않는다. **동일 청크 판정 키**: `(문서 식별자 + 청크 내용 해시)`를 기준으로 한다. 동일 키가 이미 인덱스에 있으면 재삽입하지 않는다. 키 산출 방식의 상세(해시 알고리즘·정규화 범위)는 → [03_DATABASE/](../03_DATABASE/) · [시작프롬프트.md](../../시작프롬프트.md) §B-2 ④에서 확정한다. [확인필요] |
| AC-5.3 | 실패 처리 | 파싱 실패 문서는 전체 중단 없이 스킵·로그된다 |

### US-6 (P0) — 검색 품질 평가 · L1

- **As a** 평가자,
- **I want** 골든셋으로 Recall@k·MRR·nDCG를 측정하고,
- **So that** 변경이 품질을 개선했는지/회귀시켰는지 판단하고 싶다.

**Acceptance Criteria**

| # | 조건 | 기대 결과 |
| --- | --- | --- |

## 5. 구체 질의 예시 시나리오

### 5.1 "연차 휴가 신청 절차" (회귀 케이스)

| 항목 | 내용 |
| --- | --- |
| 질의 | `연차 휴가 신청 절차` |
| 기대 정답 | 휴가규정 문서의 신청 절차 청크 |
| 대표 오답 | 출장경비 규정 문서 |
| 베이스라인 결과 | 정답 **3위**, 오답(출장경비)보다 점수 격차 **-0.018** → **Recall@1=0** |
| 의미 | nomic 임베딩의 한국어 약점이 드러나는 1급 회귀 케이스 |

이 시나리오는 시스템의 존재 이유(한국어 검색 품질)를 가장 압축적으로 보여줍니다. 개선 후보 **BGE-M3**(1024차원, ctx 8192, Dense+Sparse+ColBERT 하이브리드, MIT 라이선스)의 효과는 바로 이 케이스로 검증합니다. → [시작프롬프트.md](../../시작프롬프트.md) §B-5(임베딩 교체 결정). GGUF 다운로드·교체 여부는 **DECISION**(사용자 결정), 효과 측정 절차는 → [05_운영_평가/](../05_운영_평가/).

### 5.2 응답 폴백이 필요한 질의 (예: 단문 질문)

세 생성모델 모두 `message.content`가 빈 문자열로 오고 답은 `reasoning_content`에 담깁니다(세 모델 공통 재현). 모델 사양은 다음과 같습니다(컨텍스트 길이는 핵심 불변식이며 무한사고 잘림 맥락과 직결).

| 모델 | 종류 | 컨텍스트 | 비고 |
| --- | --- | --- | --- |
| `qwen/qwen3.6-35b-a3b` | 35B MoE | **8192** | ctx 좁음 → 청크 수 제한 |
| `qwen/qwen3.5-9b` | 9B급 `[추정]` | **8192** | 경량·빠름, ctx 좁음 |
| `google/gemma-4-e4b` | 4B급 `[추정]` | **65536** | ctx 넓음 → 긴 컨텍스트 RAG 유리, 소형이라 품질 한계 가능 |

> 모델 사양 정본은 → [시작프롬프트.md](../../시작프롬프트.md) §B-3. 파라미터 수 9B/4B는 모델명 유추값이라 `[추정]`이고, ctx(8192/8192/65536)와 35B MoE는 실측입니다. 공통 설정: 모두 로컬, `parallel=4`.

따라서 한 문장 질의에도 아래 폴백 시퀀스가 적용되어야 합니다(정본 코드 → [시작프롬프트.md](../../시작프롬프트.md) §B-4 · §B-7).

```text
응답 수신
  └─ content 비어있음?
       ├─ 예 → reasoning_content 사용 → <think> 블록 제거 → 반환
       └─ 아니오 → content 반환
  └─ finish_reason == length?
       └─ 예 → 더 높은 max_tokens로 1회 재시도(잘림 방지)
```

> 무효로 확인된 우회책(실측): `/no_think`, `enable_thinking=false`, `reasoning_effort=low`, `max_tokens=4000` 전부 효과 없음. BLOCKING 상세·근본 대응은 → [시작프롬프트.md](../../시작프롬프트.md) §B-4.
