# 모듈 설계

src/ 하위 6개 모듈(llm·ingest·embed·store·retrieve·app)의 책임·입출력·의존방향을 단일책임 원칙으로 정의하고 디렉토리에 매핑합니다.

> 관련: [../../시작프롬프트.md](../../시작프롬프트.md) · [00_시스템아키텍처.md](00_시스템아키텍처.md) · [02_내부API_인터페이스.md](02_내부API_인터페이스.md)

---

## 1. 설계 원칙

- **단일책임(SRP)**: 모듈 하나당 책임 하나. LMStudio 통신·청킹·임베딩·저장·검색·답변은 서로 다른 모듈입니다.
- **단방향 의존(상위→하위)**: 의존은 항상 위에서 아래로만 흐릅니다. 하위 모듈은 상위 모듈을 import 하지 않습니다(순환 의존 금지).
- **얇은 파이프라인**: 자체 구현은 최소화하고, 외부 라이브러리(openai SDK·kss·Kiwi 등) 위에 얇은 어댑터만 둡니다.
- **불변식 게이트는 한 곳에서**: norm 정규화 가드, reasoning 폴백 등 docs-em 실측 불변식은 해당 책임 모듈 내부에서만 처리하고 다른 모듈로 새어 나가지 않게 합니다.

의존 계층(위가 상위, 아래가 하위)은 다음과 같습니다.

```
app  ─────────────► retrieve ──► store
 │                     │           ▲
 │                     ▼           │
 └────► (질의 임베딩) embed ───────┘
            │
            ▼
           llm  ◄───────── ingest (문서 적재 시)
```

- `app`은 `retrieve`(검색)와 `embed`(질의 임베딩)·`llm`(답변 생성)에 의존합니다.
- `retrieve`는 `store`(인덱스 조회)에 의존합니다.
- `embed`는 `llm`(임베딩 API)에 의존합니다.
- `ingest`는 적재 파이프라인에서 `embed`·`store`를 호출하지만, 그 자체는 텍스트 산출까지만 책임집니다.
- `llm`은 어떤 상위 모듈도 import 하지 않는 최하위 어댑터입니다.

## 2. 모듈별 책임

### 2.1 `src/llm` — LMStudio 래퍼

| 항목 | 내용 |
|------|------|
| 책임 | 로컬 LMStudio(`http://localhost:1234/v1`, OpenAI 호환) 통신의 단일 진입점. `chat`(생성)·`embed`(임베딩) 두 함수 제공. |
| 입력 | `chat`: messages·model·max_tokens 등 / `embed`: 텍스트 리스트·model |
| 출력 | `chat`: 정제된 답변 문자열 / `embed`: 768차원 float 벡터 리스트 |
| 의존 | openai SDK(`base_url` 교체, `api_key='lm-studio'` 더미). 상위 모듈 의존 없음. |

**이 모듈이 흡수하는 불변식**:

- **reasoning 폴백**: 세 생성모델 모두 `message.content`가 빈 문자열로 오고 답이 `message.reasoning_content`로 옵니다. `content`가 비면 `reasoning_content`로 폴백하고 `<think>` 태그를 정제합니다. `finish_reason=length`면 `max_tokens*3`으로 1회 재시도하고, 2회째도 length면 경고를 부착합니다. → [시작프롬프트.md §B-2 참조](../../시작프롬프트.md)
- **norm 가드**: 임베딩 출력은 이미 L2 정규화되어 있습니다(norm=1.0). 재정규화하지 않으며, `abs(norm-1.0)<1e-3` 가드만 둡니다. 위반 시 경고합니다.
- 외부 클라우드 API(OpenAI/Anthropic/Cohere) 호출은 절대 없습니다. `base_url`만 로컬을 가리킵니다.

### 2.2 `src/ingest` — 문서 적재·청킹

| 항목 | 내용 |
|------|------|
| 책임 | 원본 문서(PDF/DOCX 등)를 읽어 텍스트로 변환(`loaders`)하고, 한국어 친화적으로 청크 분할(`chunker`). |
| 입력 | 파일 경로 또는 디렉토리 |
| 출력 | 청크 리스트(텍스트 + 메타데이터: 출처·위치 등) |
| 의존 | pymupdf(PDF)·python-docx(DOCX), kss(문장분리), Kiwi(형태소). 적재 파이프라인에서 하위로 `embed`·`store` 호출. |

**이 모듈이 흡수하는 불변식**:

- **한국어 청킹**: 문장 분리는 kss, 토큰화는 Kiwi를 사용합니다.
- **문자 기준 입력 가드**: 청크 크기/오버랩은 토큰이 아니라 **문자 기준**으로 계산하고, 빈 입력·과대 입력에 대한 가드를 둡니다.

### 2.3 `src/embed` — 임베딩 생성

| 항목 | 내용 |
|------|------|
| 책임 | 텍스트 리스트를 임베딩 벡터로 변환. `llm.embed`를 호출하는 얇은 래퍼이되, 배치 한계 분할을 담당. |
| 입력 | 텍스트 리스트 |
| 출력 | 768차원(현재) 벡터 리스트 |
| 의존 | `src/llm` |

**이 모듈이 흡수하는 불변식**:

- **배치 한계 분할**: 입력 텍스트 수가 API/메모리 한계를 넘으면 배치로 나눠 호출하고 결과를 이어 붙입니다.
- 차원은 현재 768(nomic-embed-text-v1.5). BGE-M3 교체 시 1024로 바뀌며 인덱스 재생성이 필요합니다(**DECISION**). → [시작프롬프트.md §B-2 참조](../../시작프롬프트.md)

### 2.4 `src/store` — 벡터 스토어

| 항목 | 내용 |
|------|------|
| 책임 | 벡터·메타데이터·BM25 코퍼스의 영속 저장과 조회. 인덱스 헤더(메타 불변식) 관리. |
| 입력 | (쓰기) 벡터+메타 / (읽기) 질의 벡터·top_k |
| 출력 | (읽기) 후보 문서 + 점수 |
| 의존 | LanceDB 또는 ChromaDB(**DECISION**). 상위 모듈 의존 없음. |

**이 모듈이 흡수하는 불변식**:

- **인덱스 헤더**: 인덱스에 `model`(임베딩 모델명)·`dim`(차원)·`normalized`(정규화 여부)·`metric`(거리척도) 메타를 함께 저장합니다. 질의 시 헤더와 질의 임베딩의 모델/차원이 일치하는지 검증해, 모델 교체(768↔1024) 후 인덱스 불일치를 조기에 차단합니다.

### 2.5 `src/retrieve` — 검색·결합·리랭킹

| 항목 | 내용 |
|------|------|
| 책임 | dense(벡터)와 BM25(희소) 검색을 수행(`search`)하고, 두 결과를 RRF로 결합(`fusion`)하며, 선택적으로 리랭킹(`rerank`). |
| 입력 | 질의 문자열·top_k |
| 출력 | 순위가 매겨진 문서 리스트 |
| 의존 | `src/store`, `src/embed`(질의 임베딩), rank_bm25, Kiwi(BM25 토큰화), [확인필요] BGE-reranker-v2-m3(L3). |

**이 모듈이 흡수하는 불변식**:

- **하이브리드**: dense 단독으로는 '연차 휴가 신청 절차' 질의에서 정답 3위·Recall@1=0이었습니다. dense+BM25를 **RRF**로 결합해 한국어 약점을 보완합니다.
- **리랭킹은 선택**: `rerank`는 L3에서 도입하는 선택 단계이며, 로컬 서빙 방식은 [확인필요]입니다. → [시작프롬프트.md §B-2 참조](../../시작프롬프트.md)

### 2.6 `src/app` — 답변·CLI

| 항목 | 내용 |
|------|------|
| 책임 | 사용자 진입점. 검색 결과를 컨텍스트로 답변 생성(`answer`)하고, 명령줄 인터페이스(`cli`) 제공. |
| 입력 | 사용자 질의(CLI 인자/표준입력) |
| 출력 | 생성된 답변(+ 출처) |
| 의존 | `src/retrieve`, `src/embed`, `src/llm`. 최상위 모듈로, 어떤 하위 모듈도 `app`을 import 하지 않습니다. |

진입점은 `app/cli.py`입니다.

## 3. 의존 방향 요약

| 모듈 | 직접 의존(하위) | 비고 |
|------|----------------|------|
| `app` | `retrieve`, `embed`, `llm` | 최상위. 진입점 `app/cli.py` |
| `retrieve` | `store`, `embed` | 하이브리드 검색·결합·리랭킹 |
| `embed` | `llm` | 배치 분할 |
| `ingest` | `embed`, `store`(적재 시) | 텍스트 산출까지가 본 책임 |
| `store` | — | 최하위(영속) |
| `llm` | — | 최하위(외부 SDK만) |

의존은 위에서 아래로만 흐르며, 역방향·순환 의존은 금지합니다.

## 4. 디렉토리 매핑

시작프롬프트 §B-11의 디렉토리 구조에 본 모듈을 매핑합니다.

```
src/
├── llm/          # 2.1 LMStudio 래퍼: chat·embed, reasoning 폴백, norm 가드
├── ingest/       # 2.2 loaders·chunker, 한국어 kss/Kiwi, 문자 기준 입력 가드
├── embed/        # 2.3 embedder, 배치 한계 분할
├── store/        # 2.4 vector_store, 인덱스 헤더(model/dim/normalized/metric)
├── retrieve/     # 2.5 search(dense+BM25)·fusion(RRF)·rerank(선택)
└── app/          # 2.6 answer·cli (진입점 app/cli.py)
scripts/
└── env_check.py  # 환경 점검
eval/
├── queries.yaml  # 골든셋(연차휴가 회귀케이스 포함)
└── evaluate.py   # Recall@k/MRR/nDCG
tests/            # 단위·통합 테스트
data/             # 원본·인덱스 데이터
```

→ 디렉토리 전체 정의는 [시작프롬프트.md §B-11 참조](../../시작프롬프트.md)

## 5. 모듈 경계와 로드맵

| 단계 | 활성 모듈 | 비고 |
|------|----------|------|
| L1 (검색 MVP) | llm·ingest·embed·store·retrieve(dense)·app | 평가 하네스(eval/) 병행 |
| L2 (하이브리드) | + retrieve(BM25·RRF), llm(reasoning 폴백 게이트), embed(모델 교체) | 768→1024 시 인덱스 재생성 |
| L3 (리랭킹·서빙) | + retrieve(rerank) | 로컬 서빙 방식 [확인필요] |

각 단계에서 모듈의 **경계와 의존 방향은 불변**이며, 모듈 내부 구현만 확장됩니다.
