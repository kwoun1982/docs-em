## 1. 응답 규칙

### 1.1 언어·톤
- 한국어 존댓말로 작성합니다.
- 이모지, 인사말, "도와드리겠습니다" 류 사교성 문구를 쓰지 않습니다.
- 결론을 먼저 제시하고 근거·주의를 뒤에 둡니다(결론→근거→주의).

### 1.2 사실·근거 표기
- 실측 수치는 **정확히** 인용합니다. 반올림·근사로 바꾸지 않습니다.
- 근거 강도에 따라 태그를 붙입니다. 실측과 추정의 경계는 시작프롬프트.md §B-3·§B-12를 따릅니다(ctx 수치 8192/8192/65536과 35B MoE는 **실측**, 파라미터 수 9B/4B만 `[추정]`).

| 태그 | 의미 | 예 |
|---|---|---|
| (없음) | 실측·정본 확인됨 | 임베딩 768차원, norm=1.0 / gemma-4-e4b ctx 65536 |
| `[추정]` | 합리적 추론, 미검증 | qwen3.5-9b 9B급 `[추정]` (파라미터 수만 추정, ctx 8192는 실측) |
| `[확인필요]` | 출처·방법 확인 안 됨 | BGE-reranker-v2-m3 로컬 서빙 방식 `[확인필요]` |
| `**DECISION**` | 사용자 결정 사항 | BGE-M3 GGUF 다운로드 `**DECISION**` |

- 모를 때는 지어내지 않습니다. "확실하지 않습니다. [출처/방법] 확인이 필요합니다"로 답합니다.

### 1.3 중복 금지 (SSOT 원칙)
- 시작프롬프트.md에 있는 내용을 복붙하지 않습니다. `→ [시작프롬프트.md](../../시작프롬프트.md) §B-N 참조` 형식으로 **정확한 §앵커**를 달아 링크합니다(서술형 라벨만 두지 않습니다).
- 실측 불변식의 단일 출처는 시작프롬프트.md입니다. 형제 문서 [01_실측제약_불변식.md](./01_실측제약_불변식.md)는 그 도출 결과이며, 다른 docs는 인용·링크만 합니다.

## 2. 코드 규칙

### 2.1 실행 가능성
- 동작하지 않는 의사코드를 "예시"로 제출하지 않습니다. 실행 가능한 코드만 작성합니다. 미정의 함수 호출(자리표시자)을 남기지 않습니다.
- 존재하지 않는 라이브러리·함수·파라미터를 쓰지 않습니다(환각 금지). 불확실하면 `[확인필요]`로 표기하고 검증 단계를 명시합니다.

### 2.2 LMStudio 클라이언트 (절대 규칙 — INV-3)
- 모든 LLM/임베딩 호출은 로컬 LMStudio 경유입니다. 외부 클라우드 API(OpenAI/Anthropic/Cohere) 호출 코드를 작성하지 않습니다.
- openai SDK를 쓰되 `base_url`을 로컬로 교체합니다. `api_key`는 더미 문자열입니다.
- API형 임베딩(예: Upstage Solar Embedding)은 이 원칙과 충돌하므로 채택은 `**DECISION**`입니다. → [시작프롬프트.md](../../시작프롬프트.md) §B-5 참조.

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:1234/v1",
    api_key="lm-studio",  # 더미. 실제 인증 없음
)
```

- 근거 → [시작프롬프트.md](../../시작프롬프트.md) §A-1 · §B-7(클라이언트 래퍼 정본). 실측 환경 상세 → [02_로컬환경_LMStudio실측.md](../01_시스템구성/02_로컬환경_LMStudio실측.md). 셋업 절차 → [03_개발환경셋업.md](../01_시스템구성/03_개발환경셋업.md).

### 2.3 임베딩 처리 (실측 기반 금지사항 — INV-1·INV-4·INV-8)
- 출력은 이미 L2 정규화되어 있습니다(norm=1.0). **재정규화 금지(INV-1).** 다시 정규화하면 이중 정규화로 벡터가 왜곡되고 일부 0벡터에서 분모 0이 됩니다.
- 차원은 768 고정입니다(현재 모델 `text-embedding-nomic-embed-text-v1.5` 기준).
- prefix(`search_document`/`search_query`)는 효과가 미미합니다(코사인 0.687→0.695, 실측). 기본 적용하되 만능으로 기대하지 않습니다(INV-8). 단, e5 계열은 `query:`/`passage:` prefix가 실제 효과가 있어 반드시 적용해야 하므로 nomic 경험을 그대로 옮기지 않습니다.
- `usage.prompt_tokens=0`은 오보고입니다(INV-4). 토큰 회계·길이 검증에 사용하지 않고, 문자 수 기준 또는 모델별 HF 토크나이저로 로컬 계산합니다.
- 상세 → [시작프롬프트.md](../../시작프롬프트.md) §B-2 ③(임베딩) · §B-12(실측 요약). 차원·스키마 상세 → [00_벡터스토어_스키마.md](../03_DATABASE/00_벡터스토어_스키마.md).

### 2.4 생성모델 reasoning 폴백 (BLOCKING 대응 — 필수, INV-2·INV-5·INV-6)

세 생성모델 모두 `message.content`가 빈 문자열이고 답이 `message.reasoning_content`로 옵니다. 모델별 경계는 다음과 같습니다: `qwen/qwen3.6-35b-a3b`(35B MoE, ctx 8192 — 둘 다 실측), `qwen/qwen3.5-9b`(9B급 `[추정]`, ctx 8192 실측), `google/gemma-4-e4b`(4B급 `[추정]`, ctx 65536 실측). `/no_think`·`enable_thinking=false`·`reasoning_effort=low`·`max_tokens=4000` 전부 무효입니다. 모든 생성 호출은 아래 3단계를 **반드시** 거칩니다(순서 고정: 폴백 → 정제 → length 처리).

1. **content 빈값 시 reasoning_content 폴백(INV-2)**: `message.content`가 비면 `message.reasoning_content`를 답으로 채택합니다. 생략 시 답변이 빈 문자열로 보입니다.
2. **`<think>` 정제(INV-5)**: 폴백 텍스트에서 `<think>...</think>` 블록을 제거해 최종 답만 남깁니다. 생략 시 사고과정 전체(한 문장 질문에 12,972자)가 사용자에게 노출됩니다.
3. **length 재시도(INV-6)**: `finish_reason == "length"`(무한사고 잘림)면 더 높은 `max_tokens`로 재시도합니다. 재시도 결과도 동일하게 정제를 거칩니다.

정본 구현은 `_clean_reasoning`(정제)과 `chat()` 래퍼입니다 → [시작프롬프트.md](../../시작프롬프트.md) §B-7. 아래는 그 정본 함수명을 사용한 실행 가능한 최소 패턴입니다(미정의 함수 호출 없음).

```python
import re

THINK_RE = re.compile(r"<think>.*?</think>", re.DOTALL)


def _clean_reasoning(text: str) -> str:
    """<think>...</think> 블록 제거 후 최종답만 반환 (§B-7 정본)."""
    return THINK_RE.sub("", text).strip()


def chat(client, model, messages, max_tokens=4000, _depth=0):
    """content 빈값 → reasoning_content 폴백 → <think> 정제 → length 재시도(1회).
    재시도는 max_tokens를 3배로 올려 1회만 수행한다(무한 재귀 방지)."""
    resp = client.chat.completions.create(
        model=model, messages=messages, max_tokens=max_tokens,
    )
    choice = resp.choices[0]
    msg = choice.message
    # 1) 폴백: content가 비면 reasoning_content 채택
    raw = msg.content or getattr(msg, "reasoning_content", "") or ""
    # 2) 정제: 폴백/정상 모두 <think> 제거
    answer = _clean_reasoning(raw)
    # 3) length 재시도: 정제 결과를 버리지 않되, 잘렸으면 1회 상향 재시도
    if choice.finish_reason == "length" and _depth == 0:
        return chat(client, model, messages, max_tokens=max_tokens * 3, _depth=1)
    return answer
```

- 이 폴백 게이트는 로드맵 L2 항목입니다(선결 Blocking). 상세 근거·근본 대응(LMStudio 프리셋의 reasoning 파서/종료 토큰 점검 `[확인필요]`) → [시작프롬프트.md](../../시작프롬프트.md) §B-4 · §B-7. 모델별 제약 → §B-3. 호출 인터페이스 계약 → [02_내부API_인터페이스.md](../04_아키텍처_API/02_내부API_인터페이스.md). 운영 점검 → [01_운영_런북.md](../05_운영_평가/01_운영_런북.md).

### 2.5 스택 사용 (미확정 주의)
- 스택은 **권장**일 뿐 미확정입니다(`**DECISION**` 영역, §B-9 결정 포인트). 코드에서 특정 라이브러리를 확정 사실처럼 단정하지 않습니다.
- 후보: Python 3.11+, openai SDK(base_url 교체), pymupdf+python-docx, kss, Kiwi, LanceDB 또는 ChromaDB, rank_bm25, RRF, BGE-reranker-v2-m3. 이 중 **BGE-reranker-v2-m3의 로컬 서빙 방식은 `[확인필요]`**(서빙 방식 미확정 — §B-2 ⑥ 참조). 벡터스토어·BM25 토큰화 등 ★ 항목 선택은 `**DECISION**`.

## 3. 큰 결정 확인 절차 (DECISION 게이트)

비가역적이거나 비용·리스크가 큰 결정은 임의로 진행하지 않고 `**DECISION**`으로 표시해 사용자에게 확인받습니다. 정본 결정 목록 → [시작프롬프트.md](../../시작프롬프트.md) §B-9.

**DECISION이 필요한 경우(예)**
- 모델 교체/다운로드: BGE-M3 GGUF(1~2GB) 다운로드, 임베딩 모델 교체(768→1024 차원, 인덱스 전체 재생성·norm 재측정 동반)
- 벡터스토어 선택: LanceDB vs ChromaDB
- 스택 라이브러리 확정(언어·문장 분리·형태소·sparse 검색 등 ★ 항목)
- BGE-reranker-v2-m3 로컬 서빙 방식 확정(`[확인필요]` → 결정)
- 골든셋 정의 변경, 평가 기준 변경 → 정본 [01_운영_런북.md](../05_운영_평가/01_운영_런북.md) 및 [시작프롬프트.md](../../시작프롬프트.md) §B-8 기준을 바꾸는 변경
- 디렉터리 구조·인터페이스 등 광범위 영향 변경

**절차**
1. 결정 지점에서 멈추고 `**DECISION**`으로 명시합니다.
2. 선택지 2~3개와 각 트레이드오프(실측 근거 우선)를 제시합니다.
3. 권장안과 그 이유를 1줄로 답니다.
4. 사용자 확정 전에는 비가역 작업(다운로드/대량 생성/삭제)을 실행하지 않습니다.

**DECISION이 불필요한 경우**: 가역적이고 국소적인 변경(단일 함수 수정, 주석, 작은 리팩토링, 문서 오타)은 바로 진행합니다.

## 4. 실측 우선 원칙 (INV-7·INV-8)

추측보다 실측을 우선합니다. "아마 될 것"으로 코드를 확정하지 않습니다.

- **측정 후 단정**: 성능·동작 주장은 측정값으로 뒷받침합니다. 미측정이면 `[추정]`/`[확인필요]`를 붙입니다.
- **변수 1개씩**: 평가·튜닝 시 한 번에 한 변수만 바꿔 효과를 분리 측정합니다(임베딩, prefix, 청크 크기 등).
- **회귀 케이스 고정**: "연차 휴가 신청 절차" 케이스(정답 휴가규정 3위, 오답 출장경비 대비 격차 -0.018, Recall@1=0)를 골든셋 회귀 케이스로 항상 포함합니다(INV-7).
- **오보고 경계**: `usage.prompt_tokens=0`(INV-4), prefix 만능 기대(INV-8) 등 알려진 오보고/착시를 신뢰하지 않습니다.
- 평가 지표(Recall@k 1/3/5/10 · MRR · nDCG)·골든셋·측정 절차 상세 → [시작프롬프트.md](../../시작프롬프트.md) §B-8. 운영상 평가·회귀 기준 정본 → [01_운영_런북.md](../05_운영_평가/01_운영_런북.md)(중복 작성 금지).

## 5. 작업 우선순위 (§A-3 순서 고정)

생성보다 검색·평가가 앞섭니다. 순서를 뒤집지 않습니다.

```
1. reasoning 폴백 (BLOCKING 해소)  →  생성모델이 답을 못 내면 그 위 모든 게 무의미
2. 검색 품질 (1순위 목표)          →  한국어 문서 검색 정확도 확보
3. 평가 하네스                     →  Recall@k/MRR/nDCG·골든셋으로 측정
4. 생성 (RAG 응답)                 →  위 3개가 갖춰진 뒤 답변 생성
```

| 순위 | 작업 | 근거 | 불변식 | 로드맵 |
|---|---|---|---|---|
| 1 | reasoning_content 폴백·`<think>`정제·length 재시도 | BLOCKING: 답이 안 나옴 | INV-2/5/6 | L2 |
| 2 | 한국어 검색 품질(임베딩 교체·하이브리드) | 1순위 목표, 현재 Recall@1=0 | INV-7/8 | L1→L2 |
| 3 | 평가 하네스(지표·골든셋·변수 1개씩) | 측정 없이 개선 불가 | INV-7 | L1 |
| 4 | RAG 생성 응답 | 검색·평가가 선행 | — | L2→L3 |
