# 실측 제약 불변식 (변경금지)

AI가 절대 위반하면 안 되는 docs-em 실측사실 체크리스트 INV-1~INV-8과 위반 시 발생하는 실패를 정의합니다.

> 관련: [시작프롬프트.md §B-12](../../시작프롬프트.md) · [00_AI작업규약.md](./00_AI작업규칙.md) · [03 임베딩 설계](../03_임베딩/) · [04 생성모델 폴백](../04_생성모델/) · [05 검색 파이프라인](../05_검색/)

---

## §0. 이 문서의 사용법

이 문서는 **불변식(invariant)** 모음입니다. 불변식이란 코드·문서·설계가 어떤 상태에 있든 항상 참이어야 하는 실측 사실입니다. AI가 본 프로젝트의 코드를 작성·수정·리뷰하거나 문서를 작성할 때, 아래 INV-1~INV-8을 **위반하는 산출물을 만들면 안 됩니다.**

각 불변식은 다음 4요소로 구성됩니다.

| 요소 | 의미 |
|------|------|
| **불변식** | 항상 지켜야 할 실측 사실 |
| **위반 시 실패** | 어기면 실제로 발생하는 구체적 장애 |
| **측정 임계값** | 가드/검증에 쓰는 정량 기준 |
| **단일 출처** | 근거가 되는 SSOT 포인터 (1곳) |

원칙: **실측은 환각으로 덮어쓰지 않습니다.** "보통 임베딩은 정규화가 필요하다" 같은 일반론으로 실측(`norm=1.0`)을 무시하면 안 됩니다.

---

## §1. INV-1 — 임베딩 재정규화 금지

- **불변식**: nomic-embed-text-v1.5 출력은 **이미 L2 정규화**되어 있습니다(`norm=1.0`). 코드에서 다시 정규화하면 안 됩니다.
- **위반 시 실패**: 정규화된 벡터를 재정규화하면 수치는 거의 같지만, "정규화가 필요하다"는 잘못된 전제가 코드에 박혀 모델 교체 시 **norm 미측정으로 잘못된 유사도**를 계산하게 됩니다. 더 위험한 것은, 정규화 안 된 모델로 교체했는데도 가드가 없으면 침묵 실패합니다.
- **측정 임계값**: `abs(norm - 1.0) < 1e-3` 가드를 **임베딩 출력 직후** 1회 검사. 위반 시 즉시 예외.
- **모델 교체 시**: BGE-M3 등으로 교체하면 768→1024 차원 변경과 함께 **norm을 재측정**해야 합니다. "nomic이 정규화돼 있었으니 BGE-M3도 그럴 것"이라는 가정 금지.
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), 임베딩 절 (`prefix 0.687→0.695`, `prompt_tokens=0`와 동일 출처)

```python
# 가드 예시 (개념)
import numpy as np
def assert_normalized(vec: np.ndarray) -> None:
    norm = float(np.linalg.norm(vec))
    if abs(norm - 1.0) >= 1e-3:
        raise ValueError(f"임베딩 norm 가정 위반: norm={norm:.6f} (모델 교체 시 재측정 필요)")
```

## §2. INV-2 — content 빈값 → reasoning_content 폴백

- **불변식**: 생성모델 3종(qwen3.6-35b-a3b, qwen3.5-9b, gemma-4-e4b) 모두 `message.content`가 **빈 문자열**로 오고, 실제 답은 `message.reasoning_content`에 담깁니다.
- **위반 시 실패**: `content`만 읽으면 **빈 응답**을 받습니다. `/no_think`, `enable_thinking=false`, `reasoning_effort=low`, `max_tokens=4000`은 **전부 무효**라 옵션으로 우회할 수 없습니다.
- **측정 임계값**: `content`가 빈 문자열(`""` 또는 공백뿐)이면 `reasoning_content`로 폴백. 둘 다 비면 오류로 처리.
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), BLOCKING 절. 상세 폴백 로직은 [04_생성모델](../04_생성모델/)에서 구현.

## §3. INV-3 — 외부 API 금지

- **불변식**: 모든 LLM·임베딩 호출은 **로컬 LMStudio**(`http://localhost:1234/v1`, OpenAI 호환, `api_key='lm-studio'` 더미)로만 합니다. OpenAI / Anthropic / Cohere 등 **외부 클라우드 API는 절대 금지**입니다.
- **위반 시 실패**: 외부 API 호출은 (1) 프로젝트 제약 위반, (2) 비용·키 유출 위험, (3) 재현 불가능한 평가를 초래합니다. 골든셋 측정의 재현성이 깨집니다.
- **측정 임계값**: `base_url`이 `localhost:1234`가 아닌 호출은 0건. 코드·예제·테스트 어디에도 외부 엔드포인트/실제 키가 없어야 합니다.
- **단일 출처**: → [시작프롬프트.md §A-0, §A-2, §B-12](../../시작프롬프트.md)

## §4. INV-4 — prompt_tokens=0 토큰 불신

- **불변식**: 임베딩 응답의 `usage.prompt_tokens`가 **0으로 오보고**됩니다. 이 값을 신뢰하면 안 됩니다.
- **위반 시 실패**: `prompt_tokens`로 입력 길이/청크 크기/컨텍스트 한계를 판단하면, 모두 0이므로 **길이 검증이 무력화**되어 모델 입력 한계 초과를 놓칩니다.
- **측정 임계값**: 토큰 한계 판단은 `usage`가 아니라 **입력 측 가드**(문자 수/토크나이저 추정)로 합니다. nomic 입력 한계는 [확인필요].
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), 임베딩 절 (`prompt_tokens=0`)

## §5. INV-5 — `<think>` 정제

- **불변식**: `reasoning_content` 폴백으로 얻은 텍스트에는 `<think>...</think>` 등 **사고 흔적이 섞여** 있을 수 있으므로 정제해야 합니다.
- **위반 시 실패**: 정제하지 않으면 사용자 노출 답변/평가 입력에 사고 토큰이 섞여 **답변 품질·평가 정확도**가 오염됩니다.
- **측정 임계값**: `<think>` 블록 제거 후, 결과가 비면 폴백 실패로 간주.
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), BLOCKING 절. 구현은 [04_생성모델](../04_생성모델/).

## §6. INV-6 — `length` 잘림 → max_tokens×3 1회 재시도

- **불변식**: 무한 사고로 `finish_reason=length`(잘림)가 발생합니다(한 문장 질문에 12,972자 사례). 잘리면 `max_tokens`를 **3배로 늘려 1회만 재시도**합니다.
- **위반 시 실패**: 재시도 없이 잘린 출력을 쓰면 **불완전한 답**을 그대로 사용합니다. 무한 재시도하면 비용·시간이 폭증합니다.
- **측정 임계값**: `finish_reason == "length"`이면 `max_tokens *= 3`로 **딱 1회** 재시도. **2회째도 length면** 경고를 부착해 반환(무한 루프 금지).
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), BLOCKING 절. 구현은 [04_생성모델](../04_생성모델/).

## §7. INV-7 — 정량 증명 (Recall@1=1 합격선)

- **불변식**: 개선은 **정량 지표로 증명**합니다. 회귀 케이스('연차 휴가 신청 절차')의 합격선은 **Recall@1 = 1**입니다.
- **현재 실측(불합격)**: '연차 휴가 신청 절차' 질의에서 정답(휴가규정)이 **3위**, 오답(출장경비)보다 격차 **-0.018** → 현재 **Recall@1 = 0**.
- **위반 시 실패**: "체감상 좋아졌다" 식 정성 주장으로 개선을 인정하면, 회귀를 놓치고 잘못된 방향으로 최적화합니다.
- **측정 임계값**: 골든셋(`eval/queries.yaml`, 연차휴가 회귀 케이스 포함)에서 Recall@k(1/3/5/10)·MRR·nDCG 측정. 임베딩 교체/리랭커 후 목표 **Recall@1 = 1**. 변수는 **1개씩** 변경해 측정.
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), 평가/한국어 약점 절

## §8. INV-8 — prefix 효과 미미

- **불변식**: nomic의 `search_document` / `search_query` **prefix 효과는 미미**합니다(`0.687 → 0.695`). prefix를 한국어 약점의 주된 해법으로 기대하면 안 됩니다.
- **위반 시 실패**: prefix 튜닝에 시간을 쏟으면, 실제 해법(임베딩 교체 BGE-M3·리랭커·하이브리드)을 미루게 됩니다.
- **측정 임계값**: prefix 변경은 +0.008 수준의 미미한 변화로만 취급. 한국어 Recall@1 개선의 주 레버는 **모델 교체/리랭커**.
- **단일 출처**: → [시작프롬프트.md §B-12](../../시작프롬프트.md), 임베딩 절 (`0.687→0.695`)

## §9. 실측 vs 추정 구분 (환각 금지선)

불변식의 강도가 출처별로 다릅니다. **실측은 단정**, **추정은 `[추정]` 태그**로만 다룹니다.

| 사실 | 강도 | 비고 |
|------|------|------|
| nomic 768차원, norm=1.0, prefix 0.687→0.695, prompt_tokens=0 | **실측** | 재측정 없이 단정 가능 |
| content 빈값 → reasoning_content, finish_reason=length(12,972자) | **실측** | BLOCKING 사실 |
| '연차 휴가' 정답 3위, 격차 -0.018, Recall@1=0 | **실측** | 회귀 케이스 |
| 생성모델 **35B MoE**, **ctx**(8192/8192/65536) | **실측** | 단정 가능 |
| 생성모델 **파라미터 수**(9B급/4B급) | **[추정]** | 9b·e4b는 라벨 기반 추정 |
| 생성모델 **품질 평가**(우열·적합도) | **[추정]** | 정량 측정 전까지 추정 |
| BGE-M3 한국어 Recall@1 우수 | **[추정/벤더주장]** | 자체 골든셋 측정으로 확정 |
| nomic 입력 토큰 한계 수치 | **[확인필요]** | 미측정 |
| 리랭커(BGE-reranker-v2-m3) 로컬 서빙 방식 | **[확인필요]** | 서빙 경로 미확정, L3 도입 |

핵심: **생성모델 품질 우열은 `[추정]`이고, 35B MoE와 ctx 수치는 실측**입니다. 이 둘을 섞어 "qwen35b가 더 좋다"를 실측처럼 단정하면 안 됩니다.

## §10. AI 자가점검 체크리스트

코드/문서 산출물을 내기 전, AI는 아래를 **스스로** 점검합니다.

- [ ] **INV-1** 임베딩 벡터를 재정규화하는 코드가 없는가? `abs(norm-1.0)<1e-3` 가드가 출력 직후 있는가? 모델 교체 시 norm 재측정을 명시했는가?
- [ ] **INV-2** `message.content` 빈값 시 `reasoning_content` 폴백이 있는가? `/no_think`·`enable_thinking=false`·`reasoning_effort` 등 무효 옵션에 의존하지 않는가?
- [ ] **INV-3** `base_url`이 `localhost:1234`인가? 외부 API/실제 키가 코드·예제·테스트에 없는가?
- [ ] **INV-4** `usage.prompt_tokens`로 길이를 판단하지 않는가? 입력 측 가드로 길이를 검증하는가? (nomic 입력 한계 수치는 [확인필요])
- [ ] **INV-5** `reasoning_content`에서 `<think>` 블록을 정제하는가?
- [ ] **INV-6** `finish_reason=length`면 `max_tokens×3` **1회** 재시도하고, 2회째 length면 경고를 부착하는가?
- [ ] **INV-7** 개선을 골든셋 정량 지표로 증명하는가? 회귀 케이스 Recall@1 합격선을 지키는가? 변수를 1개씩 바꿨는가?
- [ ] **INV-8** prefix를 한국어 약점의 주 해법으로 과대평가하지 않는가?
- [ ] **추정/미확인** 생성모델 품질 우열을 `[추정]` 없이 단정하지 않는가? 리랭커 서빙 방식을 [확인필요]로 표기했는가?

## §11. 형제 문서 enforcement 위치 매핑

각 불변식이 **어디서 코드로 강제(enforce)되는지**를 매핑합니다. 본 문서는 "무엇을"이고, 형제 문서는 "어떻게"입니다.

| 불변식 | enforcement 위치 (형제 문서) | 비고 |
|--------|------------------------------|------|
| INV-1 재정규화 금지·norm 가드 | [03 임베딩 설계](../03_임베딩/) | 임베딩 출력 가드, 모델 교체 재측정 |
| INV-4 prompt_tokens 불신·입력 가드 | [03 임베딩 설계](../03_임베딩/) | 입력 측 길이 가드 |
| INV-8 prefix 효과 | [03 임베딩 설계](../03_임베딩/) | prefix 실험 결과 기록 |
| INV-2 content 폴백 | [04 생성모델 폴백](../04_생성모델/) | 폴백 게이트 구현 |
| INV-5 `<think>` 정제 | [04 생성모델 폴백](../04_생성모델/) | 정제 함수 |
| INV-6 length 재시도 | [04 생성모델 폴백](../04_생성모델/) | 재시도 로직 |
| INV-7 정량 증명·Recall@1 | [05 검색 파이프라인](../05_검색/) · `eval/evaluate.py` | 골든셋·평가 하네스 |
| INV-3 외부 API 금지 | 전 영역 (`scripts/env_check.py`) | LMStudio 연결 점검 |

## §12. 요약

- 불변식 8개는 **실측에서 도출**되었고, 변경하려면 **재측정**이 선행되어야 합니다.
- **실측(norm=1.0, 35B MoE, ctx, Recall@1=0, 12,972자)은 단정**, **품질 우열은 `[추정]`**, **리랭커 서빙·nomic 입력 한계는 `[확인필요]`**.
- AI는 산출물 제출 전 §10 체크리스트를 통과해야 하며, 실제 강제 위치는 §11 형제 문서에 있습니다.
