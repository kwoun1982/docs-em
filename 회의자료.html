한 줄 요약: 우리 회사의 PDF·워드 문서를 컴퓨터에 넣어두고, "연차 휴가 어떻게 신청해?"처럼 평소 말로 물으면 관련 문서를 정확히 찾아 답해 주는 AI를 만듭니다. 핵심은 외부 인터넷·외부 AI(챗GPT 등)를 전혀 쓰지 않고 우리 컴퓨터 안에서만 돌린다는 점입니다(보안·비용).

1

이 프로젝트가 뭔가요? (개요)

함께 이해하기

회사에는 휴가규정, 출장경비 규정 같은 문서가 PDF·워드로 잔뜩 쌓여 있습니다. 사람들이 필요한 내용을 일일이 뒤지기 어렵습니다. 그래서 "평소 말로 물어보면 알맞은 문서를 찾아 답해 주는 AI"를 만듭니다. 이런 걸 RAGRAG (검색 기반 답변)
AI가 그냥 머리로 지어내는 게 아니라, 먼저 우리 문서에서 근거를 찾아온 뒤 그걸 보고 답하는 방식. 그래서 거짓말(환각)이 줄고 출처를 댈 수 있습니다. 시스템이라고 부릅니다.

🟢 쉽게 말하면

회사 문서 전용 "똑똑한 검색창"입니다. 네이버처럼 검색하되, 대상이 우리 회사 내부 문서이고, 게다가 답까지 정리해 줍니다. 단 인터넷에 안 보내고 우리 컴퓨터 안에서만 돌립니다.

왜 만드냐면 — 지금 문제가 있습니다. 시험 삼아 "연차 휴가 신청 절차"라고 물었더니, 정답인 휴가규정 문서가 1등이 아니라 3등으로 나왔습니다. 엉뚱한 출장경비 문서보다도 살짝 뒤로 밀렸습니다. 사람 눈엔 뻔한 정답을 AI가 1등으로 못 올린 겁니다. 이걸 고치는 게 목표입니다.

어떤 구조로 만들 것인가 (오늘의 핵심)

오늘 메인 안건

이 AI는 두 단계로 일합니다. (1) 미리 문서를 정리해 두는 준비 작업과, (2) 사람이 물을 때 실제로 찾아 답하는 실시간 작업입니다.

🟢 쉽게 말하면

도서관 비유입니다. ① 준비 작업 = 사서가 미리 모든 책을 읽고 색인 카드를 만들어 서가에 정리해 두는 일. ② 실시간 작업 = 손님이 질문하면 그 색인을 보고 딱 맞는 책을 꺼내 요약해 주는 일.

① 준비 작업
(미리 해둠) 문서 PDF/워드→ 글자 뽑아내기→ 잘게 쪼개기→ 숫자로 바꾸기→ 창고에 저장
② 실시간 작업
(질문 시) 질문→ 관련 문서 찾기→ 순위 매기기→ 한번 더 정렬(선택)→ AI가 답 정리→ 답변+출처

오늘 정할 4가지 "부품 선택"입니다. 각각 안건으로 따로 다룹니다.

3

"검색 두뇌"를 더 좋은 걸로 바꿀까?

바꾸되, 좋아질 때만 채택

문서를 검색하려면 글을 숫자(벡터)임베딩 / 벡터
글의 의미를 숫자 목록으로 바꾼 것. 의미가 비슷한 글은 숫자도 비슷해집니다. 그래서 "휴가"와 "연차"처럼 단어가 달라도 가까운 걸로 찾아낼 수 있습니다. 로 바꿔야 합니다. 이 변환을 하는 부품이 검색 두뇌(임베딩 모델)인데, 지금 쓰는 게 한국어에 약합니다. 그래서 더 잘한다는 후보(BGE-M3)로 바꾸는 걸 검토합니다.

🟢 쉽게 말하면

지금 두뇌는 영어는 잘하는데 한국어는 좀 둔합니다. 한국어를 더 잘한다는 새 두뇌가 있어서 갈아 끼우려 합니다. 단 "진짜 우리 문서에서 더 잘 맞히는지" 직접 시험해 보고, 좋아질 때만 바꿉니다. 광고만 믿고 바꾸지 않습니다.

바꾸면 드는 일—기존에 저장해 둔 문서를 전부 다시 정리해야 함

바꾼 숫자를 어느 "창고"에 저장할까?

둘 중 하나 정하기

문서를 숫자로 바꾼 뒤, 그걸 빠르게 꺼내 쓰려면 전용 저장소가 필요합니다. 이걸 벡터 창고벡터 저장소(Vector Store)
의미를 숫자로 바꾼 데이터를 저장하고, "이 질문과 가장 비슷한 문서"를 빠르게 찾아주는 특수 데이터베이스. 별도 서버 없이 그냥 파일로 동작합니다. 라고 합니다. 후보가 둘인데, 둘 다 서버 없이 그냥 파일로 동작하고 인터넷도 안 씁니다.

🟢 쉽게 말하면

책 보관 창고를 두 업체 중 고르는 일입니다. 둘 다 우리 건물 안에 짓고(외부 안 씀) 잘 작동합니다. 권장은 공간을 적게 먹고 분류가 빠른 쪽(LanceDB)입니다.

답을 써줄 "AI 작가"를 누구로 할까?

고르되, 나중에 검증

문서를 찾아온 뒤, 그걸 읽고 사람이 보기 좋게 답을 정리해 주는 AI를 골라야 합니다. 우리 컴퓨터에 이미 후보 2개가 깔려 있습니다.

🟢 쉽게 말하면

두 명의 작가 중 하나를 고릅니다. 한 명(gemma)은 한 번에 아주 많은 문서를 읽을 수 있고, 다른 한 명(qwen)은 글솜씨(추론)가 더 낫지만 한 번에 읽는 양이 적습니다. 검색엔 많이 읽는 쪽이 유리해서 그쪽을 권장합니다.

검색 방식 두 개를 합쳐서 더 정확하게

방법 공유

검색에는 두 가지 방식이 있고, 우리는 둘을 합쳐서 씁니다.

  • 의미로 찾기 — "휴가"라고 물으면 "연차"가 든 문서도 찾아냄 (뜻이 비슷하니까)
  • 단어로 찾기 (BM25)BM25 / 키워드 검색
    옛날 검색처럼 똑같은 단어가 들어있는지로 찾는 방식. 정확한 용어(예: 사번, 규정 번호)를 찾을 땐 의미 검색보다 강합니다.     — 질문에 든 단어가 똑같이 들어간 문서를 찾음 (정확한 용어에 강함)
🟢 쉽게 말하면

한 명은 "뜻이 비슷한 책"을 잘 찾고, 다른 한 명은 "제목에 그 단어가 박힌 책"을 잘 찾습니다. 둘의 추천을 공정하게 합쳐서RRF (순위 합치기)
두 검색이 각자 매긴 순위를 공정하게 합산해 최종 순위를 내는 방법. 한쪽이 1등이라고 한 걸 양쪽이 다 높게 본 걸 위로 올립니다. 최종 추천 목록을 만듭니다. 한쪽만 쓰는 것보다 정확합니다.

논의: "단어로 찾기"를 넣었을 때 실제로 점수가 오르는지 켜고/끄고 비교해 본 뒤 채택합니다. 2단계

"지식 지도(온톨로지)"로 한국어 약점 직접 보완하기

신규 안건 · 논의

지금까지(안건 3·6)는 AI가 통계로 알아서 의미를 찾게 하는 방식이었습니다. 그런데 한국어 동의어는 AI가 자주 헷갈립니다. 그래서 사람이 직접 만든 지식 지도를 더해 보완합니다. 이걸 온톨로지온톨로지 (지식 지도)
"우리 회사 문서에 어떤 개념이 있고, 서로 어떤 관계이며, 같은 뜻의 다른 말이 뭔지"를 사람이 직접 정리해 둔 지식 지도. AI의 통계적 추론과 달리 명시적 규칙입니다. 라고 합니다. AI 방식을 대체하는 게 아니라 같이 씁니다.

🟢 쉽게 말하면

AI에게 "우리 회사 용어집과 조직도를 미리 쥐여주는" 겁니다. AI 혼자선 "연차"와 "연가"가 같은 말인지 헷갈리지만, 사람이 만든 용어집에 "연차=휴가=연가"라고 적어두면 확실히 같이 찾습니다. 회귀 케이스(연차 휴가 3등) 같은 문제에 가장 직접적인 약점 보완입니다.

지식 지도는 세 부분으로 구성됩니다.

① 동의어 사전

연차 = 휴가 = 연가 = 월차
가장 싸고 효과 큼 · 2단계

② 관계 그래프

휴가규정 → 관할 → 인사팀
관계 따라 보강 · 3단계

③ 분류 체계

인사/근태 ↔ 재무/경비
엉뚱한 주제 걸러내기 · 2~3단계

임베딩과의 관계

대체 아님, 보완
AI 그물 + 사람 조준
구분임베딩 (AI 자동)온톨로지 (사람 정의)
만드는 법AI가 학습사람이 용어집·조직도 작성
강점처음 보는 표현도 의미로 매칭정확한 동의어·관계·분류
약점한국어 동의어·희귀 용어 약함사전에 없으면 못 잡음·관리 필요
논의 포인트:
① 동의어 사전부터 만들자 (YAML 파일만 쓰면 되어 비용 최소, 한국어 약점 직격). 2단계
② 효과는 역시 켜고/끄고 점수 비교로 확인 (임베딩 교체와 동시에 켜지 않기).
③ 관계 그래프는 만드는 비용이 커서 맨 나중(3단계).
④ 용어집·조직도를 누가 작성·관리할지 정해야 함 (잘못된 동의어는 오히려 검색을 망침).
※ 온톨로지 데이터·추론도 전부 우리 컴퓨터 안에서만 (외부 인터넷 금지 원칙 유지).

검색 결과를 한 번 더 꼼꼼히 정렬하기 (나중에)

도입 예정 · 후반 작업

1차로 추려낸 후보들을, 더 깐깐한 채점관에게 한 번 더 보여 순서를 다시 매기는 단계입니다. 이걸 리랭커리랭커 (재정렬기)
빠른 검색으로 후보 10개쯤 추린 뒤, 더 정밀하지만 느린 모델로 그 10개만 다시 꼼꼼히 순위 매김. 속도와 정확도를 절충하는 방법입니다. 라고 합니다. 후반(3단계) 작업이라 지금 당장은 아닙니다.

🟢 쉽게 말하면

1차 검색이 후보 10권을 빠르게 골라오면, 전문 심사위원이 그 10권만 다시 정독해서 진짜 1등을 가려냅니다. 전체를 다 정독하면 느리니까, 추려진 것만 다시 봅니다.

도입 전 확인할 점

  • 이 심사위원을 우리 컴퓨터 안에서 돌릴 방법이 되는지 먼저 확인해야 함 (외부 인터넷 금지 원칙 유지).
  • 실제로 점수가 오르는지 확인하고, 안건 3의 두뇌 교체와 동시에 켜지 않기 (뭐가 효과인지 헷갈리니까).

지금 AI가 말썽을 부립니다 (꼭 고쳐야 함)

반드시 해결 · 합의 필요

답변용 AI 3개 모두 이상하게 동작합니다. 답을 정상 칸에 안 넣고 "혼잣말(생각 과정) 칸"에 넣어 보냅니다. 게다가 생각을 안 멈춰서, 한 줄짜리 질문에 12,972자를 쏟아내다 중간에 잘립니다.

🟢 쉽게 말하면

직원에게 "답만 적어줘" 했더니, 답란은 비워두고 메모지(혼잣말)에 끄적인 걸 제출하는 격입니다. 게다가 메모를 끝없이 적다가 종이가 모자라 잘립니다. 그래서 우리가 "메모지라도 받아서, 결론만 깔끔히 추려내는" 처리를 코드에 넣어야 합니다.

합의 사항

  • 답란이 비면 혼잣말 칸에서 결론을 뽑아내는 처리를 기본으로 넣는다 (AI를 바꿔도 이건 유지).
  • 답이 잘리면 딱 한 번만 더 길게 재시도한다 (무한 반복 금지).
  • 이미 시도해 봤지만 효과 없던 우회법 4가지는 쓰지 않는다 (실제로 안 통함을 확인함).
합의: 이 말썽 처리를 한 곳(공통 부품)에 모아두어, 나머지 코드가 신경 안 쓰게 한다. 1단계

"좋아졌다"를 어떻게 증명할까? (점수표)

방법 합의

모든 부품 교체의 "채택 여부"가 여기서 갈립니다. 느낌이 아니라 점수로 판단합니다. 그래서 정답이 정해진 시험 문제집을 만들어 두고, 부품을 바꿀 때마다 채점합니다.

🟢 쉽게 말하면

모의고사 채점표입니다. "연차 휴가" 같은 질문 + 정답 문서를 미리 정해두고, AI가 정답을 몇 등으로 찾아내는지 점수를 매깁니다. 목표는 그 회귀 케이스에서 3등 → 1등으로 올리는 것.

무엇을 보나뜻
Recall@1Recall@1 (정확히 1등으로 맞혔나)
정답 문서를 맨 위(1등)로 올렸으면 1점, 못 올리면 0점. 우리의 핵심 목표 지표입니다.		정답을 1등으로 올렸는가 (지금 0 → 목표 1)
변수 1개씩부품은 한 번에 하나만 바꿔 효과를 정확히 잰다
논의: 시험 문제집(골든셋)을 누가, 몇 개로 만들지 정합니다. 1단계

무엇부터 만들까 (진행 순서)

실행 합의
🟢 쉽게 말하면

1단계: 일단 검색이 되게 → 2단계: 더 똑똑하게 → 3단계: 더 정밀하게 + 자동화. 작게 시작해서 점수를 보며 키웁니다.

단계목표무엇을 함
1단계일단 검색되게 + 채점표기본 검색 + 답변 + 안건 8 말썽 처리 + 점수표 만들기
2단계더 똑똑하게검색 두뇌 교체(안건 3) + 검색 두 방식 합치기(안건 6) + 동의어 사전(안건 6+ ①)
3단계더 정밀 + 자동화결과 재정렬(안건 7) + 관계 그래프(안건 6+ ②) 도입 + 운영 자동화
오늘 결론으로 가져갈 것:
① 저장 창고 = LanceDB로 확정
② 답변 AI = gemma로 시작, 품질은 나중에 점수로 검증 · 말썽 처리는 필수
③ 새 검색 두뇌(BGE-M3)는 받아서 시험하되, 점수 오를 때만 채택
④ 지식 지도(온톨로지) 도입 — 동의어 사전부터(2단계), 관계 그래프는 나중(3단계) · 용어집 관리자 지정 필요
⑤ 결과 재정렬기는 마지막 단계에서
⑥ 다음 할 일 = 1단계(기본 검색 + 채점표) 만들기 시작

용어 사전 — 어려운 말 모음

헷갈리면 여기
RAG (검색 기반 답변)
AI가 지어내지 않고, 먼저 우리 문서에서 근거를 찾아온 뒤 그걸 보고 답하는 방식. 거짓말이 줄고 출처를 댈 수 있음.
임베딩 / 벡터
글의 의미를 숫자 목록으로 바꾼 것. 뜻이 비슷하면 숫자도 비슷해져서, "휴가↔연차"처럼 단어가 달라도 찾아냄.
벡터 저장소
숫자로 바꾼 문서를 저장하고 "가장 비슷한 문서"를 빠르게 찾아주는 특수 창고. 서버 없이 파일로 동작.
BM25 (키워드 검색)
질문에 든 단어가 똑같이 들어간 문서를 찾는 옛날식 검색. 정확한 용어·번호 찾기에 강함.
온톨로지 (지식 지도)
"어떤 개념이 있고, 서로 어떤 관계이며, 같은 뜻의 다른 말이 뭔지"를 사람이 직접 정리한 지식. AI 추론과 달리 명시적 규칙.
동의어 사전
"연차=휴가=연가"처럼 같은 뜻 단어를 한 묶음으로 정리한 표. 한국어 동의어 약점을 직접 보완.
관계 그래프 (GraphRAG)
"휴가규정 → 관할 → 인사팀"처럼 개념 사이 관계를 선으로 이은 지도. 검색·답변이 관계를 따라가게 함.
하이브리드 / RRF
"의미로 찾기"와 "단어로 찾기"의 추천을 공정하게 합쳐 최종 순위를 만드는 것. 한쪽만 쓰는 것보다 정확.
리랭커 (재정렬기)
빠르게 추린 후보 몇 개만 더 깐깐한 모델로 다시 순위를 매김. 속도와 정확도의 절충.
생성 모델 (LLM)
찾아온 문서를 읽고 사람 보기 좋게 답을 써주는 AI. 챗GPT 같은 것의 우리 컴퓨터 버전.
로컬 (Local)
우리 컴퓨터 안에서만 돈다는 뜻. 외부 인터넷·외부 AI로 데이터를 보내지 않음 (보안·비용 이점).
Recall@1
정답 문서를 맨 위(1등)로 올렸으면 성공. 우리의 핵심 목표 점수 (지금 0 → 목표 1).
골든셋 (시험 문제집)
"질문 + 정답 문서"를 미리 정해둔 채점용 모의고사. 부품을 바꿀 때마다 이걸로 점수를 잼.
청킹 (잘게 쪼개기)
긴 문서를 문단·문장 단위로 잘게 나누는 것. 통째로 넣으면 검색이 뭉툭해지므로 적당히 쪼갬.
L1 / L2 / L3 (단계)
개발 단계. 1단계=일단 되게, 2단계=더 똑똑하게, 3단계=정밀+자동화.
